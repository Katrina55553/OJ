# 消息积压与可靠性保障 — 结合 OJ 判题系统

## 面试怎么答

> "本项目使用 RabbitMQ 做异步判题，通过手动 ACK + 死信队列保障消息可靠性。生产者发送消息后，消费者判题完成才确认。如果消费者崩溃，消息自动回到队列重新投递。超过重试次数的消息进入死信队列，做兜底处理。"

---

## 1. 本项目的 MQ 架构

```
生产者（QuestionSubmitServiceImpl）
    │  发送提交 ID
    ▼
┌─────────────────────────────┐
│  oj.judge.queue（正常队列）   │
│  手动 ACK，prefetch=1        │
│  TTL: 30 秒                  │
│  死信队列: oj.judge.queue.dlq│
└─────────────┬───────────────┘
              │
              ▼
消费者（JudgeMessageConsumer）
    │  判题完成 → basicAck
    │  判题异常 → basicNack → 进入死信队列
    ▼
┌─────────────────────────────┐
│  oj.judge.queue.dlq（死信）  │
└─────────────┬───────────────┘
              │
              ▼
死信消费者 → 记录日志 → 人工处理
```

## 2. 消息可靠性：三个环节保障

### 环节 1：生产者 → 队列（消息不丢）

```java
// JudgeMessageProducer.java
@Component
public class JudgeMessageProducer {
    @Resource
    private RabbitTemplate rabbitTemplate;

    public void sendJudgeMessage(long questionSubmitId) {
        // convertAndSend 默认是确认模式
        // 消息到达交换机会回调 confirmCallback
        rabbitTemplate.convertAndSend("oj.judge.queue",
                String.valueOf(questionSubmitId));
    }
}
```

**保障**：RabbitMQ 的 `publisher confirm` 机制，消息到达交换机后回调确认。

### 环节 2：队列本身（消息持久化）

```java
// RabbitMQConfig.java
@Bean
public Queue judgeQueue() {
    return QueueBuilder.durable("oj.judge.queue")  // ⭐ 持久化队列
        .withArgument("x-dead-letter-exchange", "")
        .withArgument("x-dead-letter-routing-key", "oj.judge.queue.dlq")
        .withArgument("x-message-ttl", 30000)
        .build();
}
```

**保障**：`durable=true`，队列元数据持久化到磁盘，RabbitMQ 重启不丢。

### 环节 3：消费者（手动 ACK）

```java
// JudgeMessageConsumer.java
@RabbitListener(queues = "oj.judge.queue")
public void onMessage(Message message, Channel channel) throws IOException {
    long deliveryTag = message.getMessageProperties().getDeliveryTag();
    try {
        judgeService.doJudge(questionSubmitId);
        channel.basicAck(deliveryTag, false);      // ⭐ 成功才确认
    } catch (Exception e) {
        channel.basicNack(deliveryTag, false, false);  // ⭐ 失败拒绝，不重新入队
    }
}
```

**保障**：消费者崩溃时，未确认的消息自动回到队列，不会丢失。

## 3. 消息积压怎么处理？

### 什么是消息积压？

```
生产速度 >> 消费速度：

每秒提交 100 个判题任务
每个判题任务需要 3 秒
每秒只能消费 1 个（prefetch=1）

→ 每秒积压 99 个消息
→ 队列越来越大
→ 用户等待时间越来越长
```

### 解决方案

| 方案 | 原理 | 适用场景 |
|------|------|----------|
| **增加消费者** | 多开几个消费者并行处理 | 最直接有效 |
| **增加 prefetch** | 每个消费者同时处理多条消息 | CPU 够用时 |
| **临时扩容** | 高峰期多开消费者，低峰期缩容 | 有自动扩缩容能力 |
| **丢弃过期消息** | 设置 TTL，超时直接进死信 | 允许部分任务超时 |

### 本项目的处理

```yaml
# application.yml
spring:
  rabbitmq:
    listener:
      simple:
        prefetch: 1  # 每次只拿 1 条消息
        retry:
          enabled: true
          max-attempts: 3       # 最多重试 3 次
          initial-interval: 3000  # 重试间隔 3 秒
```

**当前配置**：单消费者、prefetch=1，适合中小规模。如果要支持高并发：
1. 水平扩展消费者（多部署几个后端实例）
2. 增大 prefetch 到 3-5
3. 考虑用线程池并行处理

## 4. 死信队列的作用

### 消息什么时候进入死信队列？

```
1. 消费者 basicNack + requeue=false  → 消息进入死信
2. 消息 TTL 过期（30 秒没被消费）    → 消息进入死信
3. 队列满了                          → 消息进入死信
```

### 死信消费者做什么？

```java
@RabbitListener(queues = "oj.judge.queue.dlq")
public void onDeadLetter(Message message, Channel channel) throws IOException {
    String body = new String(message.getBody());
    log.error("判题消息进入死信队列: submitId={}", body);

    // 更新提交状态为失败
    long submitId = Long.parseLong(body);
    QuestionSubmit update = new QuestionSubmit();
    update.setId(submitId);
    update.setStatus(QuestionSubmitStatusEnum.FAILED.getValue());
    questionSubmitService.updateById(update);

    channel.basicAck(message.getMessageProperties().getDeliveryTag(), false);
}
```

**作用**：异常兜底，保证不会有一个提交永远卡在 WAITING 状态。

## 5. 面试常见追问

### Q: 为什么用 RabbitMQ 而不是 Kafka？

> - **RabbitMQ**：消息确认机制成熟，支持死信队列、延迟队列，适合业务消息（本项目）
> - **Kafka**：高吞吐、适合日志收集、大数据场景
> - 判题任务量不大（每秒几十到几百），RabbitMQ 完全够用

### Q: 手动 ACK 和自动 ACK 的区别？

> - **自动 ACK**：消费者收到消息就确认。如果消费者处理到一半崩溃，消息丢失
> - **手动 ACK**：消费者处理完才确认。崩溃时消息回到队列，重新投递
> - 本项目用手动 ACK，保证判题任务不丢

### Q: prefetch=1 有什么好处？

> 判题是 CPU 密集型操作。prefetch=1 保证一个消费者同时只处理一个判题任务，避免 CPU 打满。如果设成 10，一个消费者同时跑 10 个 Docker 容器，CPU 直接爆了。

### Q: 消息重复消费怎么办？

> 本项目目前没有做幂等处理。理论上手动 ACK + prefetch=1 不会重复消费。如果要做幂等，可以在消费前检查提交状态：如果已经不是 WAITING 状态，直接跳过。
