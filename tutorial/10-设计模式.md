# 设计模式在 OJ 项目中的应用 — 策略模式、工厂模式、代理模式

## 面试怎么答

> "本项目在三个地方使用了设计模式：判题模块用**策略模式**封装不同语言的判题逻辑；代码沙箱用**工厂模式 + 配置化**实现多种沙箱的灵活切换；日志增强用**代理模式**统一记录沙箱调用的入参、耗时和结果。"

---

## 1. 策略模式 — 判题逻辑

### 解决什么问题？

不同语言的判题规则不同：

| 语言 | 特殊逻辑 |
|------|----------|
| C++/Python/Go/JS | 只比对输出 |
| Java | JVM 启动需要额外 10 秒，需要时间补偿 |
| Java | 需要额外检查内存限制 |

如果用 if-else：

```java
// ❌ 每加一种语言就要改这个方法
if ("java".equals(language)) {
    // Java 特殊逻辑...
} else if ("python".equals(language)) {
    // Python 特殊逻辑...
} else if ("cpp".equals(language)) {
    // C++ 特殊逻辑...
}
// 违反开闭原则：对扩展开放，对修改关闭
```

### 策略模式怎么解决？

```
┌─────────────────────────────────────────┐
│           JudgeStrategy（接口）           │
│         + doJudge(JudgeContext)          │
└──────────────────┬──────────────────────┘
                   │
      ┌────────────┴────────────┐
      ▼                         ▼
┌───────────────────┐  ┌──────────────────────────┐
│DefaultJudgeStrategy│  │JavaLanguageJudgeStrategy   │
│ - 比对输出          │  │ - 比对输出                 │
│ - 判断答案          │  │ - 内存限制检查             │
│                   │  │ - Java 启动时间补偿 10s    │
└───────────────────┘  └──────────────────────────┘
```

### 核心代码

```java
// 1. 策略接口
public interface JudgeStrategy {
    JudgeInfo doJudge(JudgeContext judgeContext);
}

// 2. 默认策略（C++/Python/Go/JS）
@Component
public class DefaultJudgeStrategy implements JudgeStrategy {
    @Override
    public JudgeInfo doJudge(JudgeContext judgeContext) {
        List<String> outputList = judgeContext.getOutputList();
        List<String> expectedOutputList = judgeContext.getExpectedOutputList();

        JudgeInfo judgeInfo = new JudgeInfo();
        judgeInfo.setTime(judgeContext.getJudgeInfo().getTime());
        judgeInfo.setMemory(judgeContext.getJudgeInfo().getMemory());

        // 逐个比对输出
        for (int i = 0; i < outputList.size(); i++) {
            if (!outputList.get(i).trim().equals(expectedOutputList.get(i).trim())) {
                judgeInfo.setMessage("Wrong Answer");
                return judgeInfo;
            }
        }
        judgeInfo.setMessage("Accepted");
        return judgeInfo;
    }
}

// 3. Java 专用策略
public class JavaLanguageJudgeStrategy implements JudgeStrategy {
    @Override
    public JudgeInfo doJudge(JudgeContext judgeContext) {
        // ... 同样的输出比对逻辑 ...

        // 额外：Java 启动时间补偿
        long JAVA_PROGRAM_TIME_COST = 10000L;  // JVM 冷启动约 10 秒
        if ((time - JAVA_PROGRAM_TIME_COST) > needTimeLimit) {
            judgeInfoResponse.setMessage("Time Limit Exceeded");
            return judgeInfoResponse;
        }

        // 额外：内存限制检查
        if (memory > needMemoryLimit) {
            judgeInfoResponse.setMessage("Memory Limit Exceeded");
            return judgeInfoResponse;
        }

        judgeInfoResponse.setMessage("Accepted");
        return judgeInfoResponse;
    }
}

// 4. 调度器选择策略
@Service
public class JudgeManager {
    JudgeInfo doJudge(JudgeContext judgeContext) {
        String language = questionSubmit.getLanguage();
        JudgeStrategy judgeStrategy = new DefaultJudgeStrategy();  // 默认
        if ("java".equals(language)) {
            judgeStrategy = new JavaLanguageJudgeStrategy();       // Java 特殊
        }
        return judgeStrategy.doJudge(judgeContext);  // 多态调用
    }
}
```

### 好处

| 好处 | 说明 |
|------|------|
| **开闭原则** | 新增语言只需加一个策略类，不改已有代码 |
| **单一职责** | 每个策略只负责一种语言的判题逻辑 |
| **可测试** | 每个策略可以独立单元测试 |

### 面试追问

**Q: 如果要加 Python 特殊策略？**

```java
// 只需新增一个类
@Component
public class PythonLanguageJudgeStrategy implements JudgeStrategy {
    @Override
    public JudgeInfo doJudge(JudgeContext judgeContext) {
        // Python 特殊逻辑：浮点数精度比对
    }
}

// JudgeManager 加一行

    judgeStrategy = new PythonLanguageJudgeStrategy();
}
```

## 2. 工厂模式 — 代码沙箱

### 解决什么问题？

代码沙箱有多种实现：

| 实现 | 用途 |
|------|------|
| `DockerCodeSandbox` | 本地 Docker 容器执行（主用） |
| `RemoteCodeSandbox` | 远程 HTTP 调用（备用） |
| `ThirdPartyCodeSandbox` | 第三方 API（备用） |

调用方不应该关心具体用哪个实现。

### 工厂模式怎么解决？

```
调用方（JudgeServiceImpl）
    │
    │  codeSandboxFactory.getSandbox()
    ▼
┌──────────────────────────┐
│   CodeSandboxFactory      │
│                          │
│   根据配置返回具体实例：    │
│   codesandbox.type=docker │
│   codesandbox.type=remote │
└──────────┬───────────────┘
           │
    ┌──────┴──────┐
    ▼             ▼
 DockerCodeSandbox  RemoteCodeSandbox
```

### 核心代码

```java
// 1. 沙箱接口
public interface CodeSandbox {
    ExecuteCodeResponse executeCode(ExecuteCodeRequest request);
}

// 2. Docker 沙箱实现
@Component
public class DockerCodeSandbox implements CodeSandbox {
    @Override
    public ExecuteCodeResponse executeCode(ExecuteCodeRequest request) {
        // 创建 Docker 容器执行代码...
    }
}

// 3. 工厂类
@Component
public class CodeSandboxFactory {
    @Resource
    private DockerCodeSandbox dockerCodeSandbox;

    @Value("${codesandbox.type:docker}")
    private String sandboxType;

    public CodeSandbox getSandbox() {
        switch (sandboxType) {
            case "docker":   return dockerCodeSandbox;
            case "remote":   return remoteCodeSandbox;
            default:         return dockerCodeSandbox;
        }
    }
}

// 4. 调用方只依赖接口
CodeSandbox codeSandbox = codeSandboxFactory.getSandbox();
ExecuteCodeResponse response = codeSandbox.executeCode(request);
```

### 配置切换

```yaml
# application.yml
codesandbox:
  type: docker    # 改这一行就能切换沙箱实现
```

### 面试追问

**Q: 为什么不直接用 @Qualifier 注入？**

> @Qualifier 是编译时绑定，改实现需要改代码重新编译。工厂模式支持**运行时动态切换**，改配置文件就行。

**Q: 还有其他沙箱实现吗？**

> 有 `RemoteCodeSandbox`（远程调用）和 `ThirdPartyCodeSandbox`（第三方 API）作为备用方案。Docker 挂了可以切到远程沙箱，实现服务降级。

## 3. 代理模式 — 日志增强

### 解决什么问题？

每次调用沙箱都要写日志：

```java
// ❌ 每个调用方都要写一遍
log.info("沙箱调用开始, language={}", language);
long start = System.currentTimeMillis();
response = codeSandbox.executeCode(request);
long cost = System.currentTimeMillis() - start;
log.info("沙箱调用结束, 耗时={}ms", cost);
```

如果有 10 个地方调用沙箱，就要写 10 遍。

### 代理模式怎么解决？

```java
/**
 * 沙箱代理，实现同一个接口，内部持有真实对象
 */
public class CodeSandboxProxy implements CodeSandbox {

    private final CodeSandbox codeSandbox;  // 被代理的真实对象

    public CodeSandboxProxy(CodeSandbox codeSandbox) {
        this.codeSandbox = codeSandbox;
    }

    @Override
    public ExecuteCodeResponse executeCode(ExecuteCodeRequest request) {
        // 前置增强
        log.info("沙箱调用开始, language={}", request.getLanguage());
        long start = System.currentTimeMillis();

        try {
            // 委托给真实对象
            ExecuteCodeResponse response = codeSandbox.executeCode(request);

            // 后置增强
            long cost = System.currentTimeMillis() - start;
            log.info("沙箱调用成功, 耗时={}ms", cost);
            return response;
        } catch (Exception e) {
            long cost = System.currentTimeMillis() - start;
            log.error("沙箱调用异常, 耗时={}ms, error={}", cost, e.getMessage());
            throw e;
        }
    }
}
```

### 使用方式

```java
// 用代理包装真实沙箱
CodeSandbox sandbox = codeSandboxFactory.getSandbox();
sandbox = new CodeSandboxProxy(sandbox);  // 代理增强

// 后续调用自动带日志
sandbox.executeCode(request);  // 自动记录入参、耗时、结果
```

### 好处

| 好处 | 说明 |
|------|------|
| **日志只写一次** | 不管多少种沙箱实现，代理统一处理 |
| **不改原有代码** | 真实沙箱类不需要加日志代码 |
| **可扩展** | 除了日志，还可以加限流、重试、监控等增强 |

### 面试追问

**Q: 代理模式和 AOP 有什么区别？**

> 本质都是代理。**代理模式**是手动包装，适合明确知道要增强哪个接口的场景。**AOP** 是声明式切面，适合按规则批量拦截（比如拦截所有 Controller 方法）。本项目两种都用了：沙箱日志用代理模式，限流和权限校验用 AOP。

## 4. 三种模式对比

| 模式 | 解决的问题 | 本项目应用 | 核心思想 |
|------|-----------|-----------|----------|
| **策略模式** | 多种算法，每种独立封装 | 判题逻辑（Default/Java） | 接口 + 多态 |
| **工厂模式** | 多种实现，运行时切换 | 代码沙箱（Docker/Remote） | 配置化创建 |
| **代理模式** | 不改原代码，增强能力 | 沙箱日志记录 | 包装 + 委托 |

---

## 5. 面试总结话术

> "我在项目中用了三种设计模式：
>
> **策略模式**用在判题模块，不同语言的判题逻辑独立封装，新增语言只需加一个策略类，不用改已有代码。
>
> **工厂模式**用在代码沙箱，通过配置文件切换 Docker 沙箱或远程沙箱，支持服务降级。
>
> **代理模式**用在沙箱调用的日志增强，统一记录入参、耗时和异常，避免在每个调用方重复写日志代码。
>
> 这三个模式分别体现了**开闭原则**、**依赖倒置原则**和**单一职责原则**。"
