## 13.1 项目一句话定位

> 这是我独立开发的一个 **全栈在线判题系统（OJ）**，类似 LeetCode，支持多语言代码提交、沙箱隔离执行、异步判题。前端 Vue 3 + Vite，后端 Spring Boot 3 + MyBatis-Plus，通过 Docker Compose 一键部署。

**关键点**：一句话覆盖「业务 + 架构 + 部署」，让面试官有继续追问的抓手。

---

## 13.2 三分钟电梯介绍模板

### 30 秒：项目背景

> 我做的是一个在线判题系统，核心场景是用户提交代码、系统在隔离环境中执行、跟预期输出比对、返回通过或失败。这类系统难点不在业务，而在 **安全** 和 **性能**——用户代码可能恶意，判题耗时可能阻塞请求。

### 60 秒：技术架构

> 整体是前后端分离架构。
>
> - **前端** Vue 3 + TypeScript + Vite，UI 用 Arco Design，代码编辑器是 CodeMirror 6。
> - **后端** Spring Boot 3 + MyBatis-Plus，MySQL 存数据，Redis 做缓存和限流，RabbitMQ 做判题异步解耦。
> - **判题核心** 是 Docker 沙箱，通过 Docker-outside-of-Docker 方式，后端容器挂载 `/var/run/docker.sock` 调用宿主机 Docker，为每条提交创建独立容器执行代码。
> - **部署** 用 Docker Compose 编排 6 个服务（MySQL/Redis/RabbitMQ/后端/前端），通过 GitHub Actions 自动部署到 VPS。

### 60 秒：技术亮点

> 我重点讲三个亮点：
>
> **第一，Docker 沙箱的 DooD 架构**。后端通过 docker socket 在宿主机创建沙箱容器，沙箱有 6 层安全限制：内存、CPU、网络禁用、PID 限制、只读文件系统、nobody 用户。每种语言独立基础镜像（cpp/java/python/go/javascript），预构建好复用。
>
> **第二，异步判题解耦**。用户提交后立即返回 submitId，判题走 RabbitMQ，消费端手动 ACK + 死信队列，避免判题失败丢消息。这样提交峰值能被 MQ 削峰，前端不卡顿。
>
> **第三，Docker 路径视角问题**。这是踩过的坑——后端容器内用 `Files.createTempDirectory` 创建的临时目录，宿主机 Docker 看不到，导致 `docker run -v` 挂载空目录，沙箱里找不到用户代码。修复方案是把 workDir 建在 docker-compose 的共享卷上（`./oj-backend/sandbox:/app/sandbox`），编译和运行都用宿主机路径挂载。

### 30 秒：工程实践

> 整个项目我坚持工程化实践：
>
> - **CI/CD**：GitHub Actions 自动跑测试、构建镜像、SSH 部署到 VPS，用 `--no-cache` 和 `--force-recreate` 保证真正更新。
> - **规范**：全局异常处理、统一响应封装、AOP 权限校验（`@AuthCheck` 注解）、Redis 滑动窗口限流。
> - **文档**：写了 13 章 ebook 教程 + tutorial 技术深度文章（Docker 沙箱、RabbitMQ、Redis 限流、JWT 等）。

## 13.3 核心技术亮点深挖

### 亮点 1：Docker 沙箱（DooD 架构）

**面试官可能问**：为什么不用 Docker-in-Docker（DinD）？

**回答**：
> DinD 需要特权容器（`--privileged`），安全风险大，而且需要在容器内跑 dockerd，资源占用高。DooD 只需挂载 `/var/run/docker.sock`，后端容器调宿主机 Docker CLI 创建兄弟容器，沙箱和后端平级，更轻量。
>
> 代价是路径视角问题：后端容器和宿主机 Docker 看到的是两套文件系统。我通过把 workDir 建在 docker-compose 共享卷上解决，这样容器内 `/app/sandbox/tmp/xxx` 和宿主机 `./oj-backend/sandbox/tmp/xxx` 指向同一个目录，`docker run -v` 挂载才能找到文件。

**面试官可能问**：6 层安全限制具体是哪些？

**回答**：

| 限制 | 参数 | 作用 |
|------|------|------|
| 内存 | `--memory=256m` | 防止 `while(true)` 吃光内存 |
| CPU | `--cpus=1` | 限制 CPU 占用 |
| 网络 | `--network=none` | 禁止访问内网/外网 |
| PID | `--pids-limit=50` | 防 fork 炸弹 |
| 文件系统 | `--read-only` | 禁止写文件系统 |
| 用户 | `--user=nobody` | 最小权限用户 |

### 亮点 2：异步判题（RabbitMQ 解耦）

**面试官可能问**：为什么用 MQ，不能同步判题吗？

**回答**：
> 同步判题有两个问题：一是阻塞——判题可能 5-10 秒，HTTP 请求一直挂着；二是并发瓶颈——单机同时跑 10 个沙箱就到资源上限。MQ 把提交和判题解耦，用户提交立即返回 submitId，前端轮询状态，MQ 消费端 prefetch=1 串行处理，避免压垮系统。
>
> 可靠性方面：手动 ACK，消费失败重试 3 次后进死信队列，不会丢消息。

**面试官可能问**：消息积压怎么办？

**回答**：
> 短期积压靠 MQ 缓冲（队列无上限），用户前端会看到 "判题中" 状态。长期方案是增加消费者数量横向扩展，或者根据机器负载动态调整 prefetch。我的 tutorial 第 9 章专门写了消息积压的分析。

### 亮点 3：Spring Boot 3 迁移踩坑

**面试官可能问**：项目用 Spring Boot 3 有什么坑？

**回答**：
> 最大的坑是 **Redis 配置前缀变更**。Spring Boot 2.x 用 `spring.redis.*`，3.x 改成 `spring.data.redis.*`。我迁移时只关注了 `javax.*` → `jakarta.*` 的包名变更，漏检了配置前缀，导致线上 Redis 连不上，应用回退默认值连 `localhost:6379`，题库接口直接系统错误。
>
> 还有几个：`mybatis-plus-boot-starter` 换成 `mybatis-plus-spring-boot3-starter`；`mysql:mysql-connector-java` 换成 `com.mysql:mysql-connector-j`；`springfox` 换成 `knife4j-openapi3-jakarta`。这些都是 Spring Boot 3 的 breaking change。

### 亮点 4：前端 Vite 迁移

**面试官可能问**：为什么从 vue-cli 迁到 Vite？

**回答**：
> 两个原因。一是开发体验——Vite 用原生 ESM，dev server 启动 <1 秒，HMR 也是毫秒级；vue-cli 用 Webpack，冷启动要几十秒。二是构建内存——VPS 只有 2GB 内存，vue-cli 构建 OOM，Vite 峰值内存低很多。
>
> 迁移后还顺便把 Monaco Editor 换成 CodeMirror 6。Monaco 在 2GB VPS 上 Vite 构建时内存峰值 2GB+，即使加 swap 也跑不过去。CodeMirror 6 依赖体积只有 Monaco 的 1/5，构建时间从 50s 降到 6s，内存峰值降到 500MB 以内。

## 13.4 常见追问及应答

### 追问 1：用户提交恶意代码怎么办？

> 三层防护。第一层是沙箱限制（前面说的 6 层）。第二层是超时机制——`docker run` 设置 `timeoutSeconds=10`，超时直接销毁容器。第三层是资源隔离——`--network=none` 让代码无法访问其他服务，`--read-only` 让代码无法写文件系统。

### 追问 2：怎么防止用户用 Java 反射逃逸？

> 当前沙箱用的是 OpenJDK + nobody 用户，反射调用受 SecurityManager 限制（虽然 Java 17 弃用了 SecurityManager，但 `--read-only` + `--network=none` + `--user=nobody` 仍能挡住大多数攻击）。生产级方案可以换成 GraalVM Native Image，或用 JVM TI agent 限制反射。

### 追问 3：判题结果怎么保证一致？

> 用例输入和预期输出都存在 MySQL 的 `judgeCase` 字段（JSON 数组）。判题时按顺序喂输入，逐个比对输出。Java 特殊处理——Java 类名必须叫 `Solution`，用 `JavaLanguageJudgeStrategy` 单独走一套逻辑（策略模式）。

### 追问 4：高并发怎么扩展？

> 横向扩展有三层。**应用层**：Spring Boot 无状态，可以加机器直接扩。**判题层**：RabbitMQ 消费者可以多实例并行消费。**数据库层**：题目读多写少，加 Redis 缓存；提交记录写多读少，可分表。当前单机够用，架构已经为扩展做好准备。

### 追问 5：为什么用 MyBatis-Plus 不用 JPA？

> MyBatis-Plus 写 SQL 更直观，复杂查询好优化；JPA 适合简单 CRUD，复杂关联要写一堆 Specification。OJ 这种场景有分页、条件查询、JSON 字段，MyBatis-Plus 的 `LambdaQueryWrapper` + 代码生成更顺手。

### 追问 6：JWT 怎么处理登出？

> JWT 是无状态的，本身不能主动失效。我的实现是 JWT 7 天过期 + Redis 缓存用户信息（TTL 30 分钟）。如果要做强制登出，可以维护一个 Redis 黑名单，`JwtInterceptor` 校验时检查 token 是否在黑名单。当前项目没做这个，因为业务场景不需要。

### 追问 7：如果让你重新设计，你会改什么？

> 三点：
>
> 1. **判题结果推送**：现在是前端轮询，改成 WebSocket/SSE 服务端推送更优雅。
> 2. **多测试用例并发执行**：现在每个用例串行 `docker run`，可以并行跑然后汇总，判题时间从 N×T 降到 T。
> 3. **沙箱复用**：现在是每次提交都创建销毁容器，可以预热一批空闲容器池，提交来了直接复用，减少容器启动开销。

## 13.6 STAR 法则答案例子

### 场景：描述一个技术难题的解决过程

**Situation（情境）**：
> 部署到 VPS 后，用户提交代码报 "python3: can't open file '/code/solution.py': No such file or directory"，但本地测试沙箱镜像和挂载都正常。

**Task（任务）**：
> 定位为什么沙箱容器里找不到用户代码文件。

**Action（行动）**：
> 第一步，在服务器手动跑 `docker run -v /tmp:/code:ro ... python3 solution.py`，输出 hello 正常，排除镜像问题。
>
> 第二步，看后端代码，发现 `DockerCodeSandbox.executeCode` 用 `Files.createTempDirectory("oj-sandbox-")` 创建临时目录，默认在容器内 `/tmp`。
>
> 第三步，意识到问题——后端容器通过 docker socket 调宿主机 Docker，`docker run -v containerPath:/code` 挂载的是后端容器内的路径，宿主机 Docker 看不到这个路径，会创建一个空目录挂载进去，所以沙箱容器里 `/code` 是空的。
>
> 第四步，修复——把 workDir 改建在 docker-compose 已挂载的共享卷上 `./oj-backend/sandbox:/app/sandbox`，容器内路径 `/app/sandbox/tmp/xxx` 对应宿主机 `./oj-backend/sandbox/tmp/xxx`，`docker run -v` 才能找到文件。同时设置 POSIX 权限 755（目录）/ 644（文件）确保 `--user=nobody` 可读。

**Result（结果）**：
> 修复后所有语言（Python/C++/Java/Go/JS）都能正常判题。这个坑写进了项目 memory，避免后续重蹈覆辙。

## 13.7 面试避坑指南

### 不要说

❌ "这是个 CRUD 项目" —— 贬低自己的项目
❌ "用了 Spring Boot" —— 太笼统，没说版本和为什么
❌ "用了 Docker" —— 没说 DooD 还是 DinD，没说为什么
❌ "前后端分离" —— 2025 年所有项目都前后端分离，不是亮点

### 要说

✅ "Docker 沙箱用 DooD 架构，后端挂载 docker.sock 调宿主机 Docker"
✅ "异步判题用 RabbitMQ 解耦，手动 ACK + 死信队列保证可靠"
✅ "Spring Boot 3 迁移踩过 Redis 配置前缀变更的坑"
✅ "VPS 2GB 内存 Vite 构建 OOM，换 CodeMirror 6 解决"
✅ "GitHub Actions 自动部署，用 --no-cache + --force-recreate 保证更新"

### 不会的别说

如果面试官问的问题你不会：

1. **诚实承认**："这个我没深入研究过，但我的理解是..."
2. **给出思路**："如果是我的话，会先查 XX 文档，然后..."
3. **转到熟悉领域**："不过我在 XX 方面有经验，可以讲讲..."

## 13.8 项目讲解节奏控制

| 时间 | 内容 | 重点 |

## 13.10 小结

面试讲项目的核心不是"展示你做了什么"，而是"展示你怎么思考的"。

| 维度 | 低分回答 | 高分回答 |
|------|---------|---------|
| 技术选型 | "用了 Spring Boot" | "用 SB 3.2，因为要 Java 17 的 record 和 sealed" |
| 难题解决 | "查了文档解决了" | "STAR 法则讲完整排查过程" |
| 架构思考 | "前后端分离" | "异步判题解耦，MQ 削峰" |
| 工程素养 | "能跑就行" | "CI/CD + 异常处理 + 限流 + 文档" |
| 学习能力 | "用了某框架" | "迁移时踩了 XX 坑，写进了 memory" |

> **最终建议**：把 13.5 的技术栈表和 13.6 的 STAR 案例背熟，面试前对着镜子讲 3 遍，语速控制在每分钟 200 字左右。面试时眼神交流 > 念稿。
