# 第 14 章 性能测试与压测

## 测试目标

量化 RabbitMQ 异步解耦相比同步判题的性能提升，用数据回答面试官：

> "引入 RabbitMQ 后，接口响应时间从 X 秒降到 Y 毫秒，吞吐量从 QPS N 提升到 QPS M。"

---

## 核心指标

| 指标 | 含义 | 测量方式 |
|------|------|----------|
| **接口响应时间** | 用户点击提交到收到"已提交"的时间 | 前端请求耗时 |
| **判题完成时间** | 从提交到出结果的总时间 | 前后端日志时间差 |
| **吞吐量（QPS）** | 每秒能处理多少个提交请求 | 压测工具统计 |
| **成功率** | 请求不报错的比例 | 压测工具统计 |
| **队列积压** | 高并发时队列中排队的消息数 | RabbitMQ 管理界面 |

---

## 测试 1：接口响应时间对比

### 目的

对比同步模式和异步模式下，用户感知的响应时间差异。

### 方法

用 `curl` 测量单次请求的响应时间：

```bash
# 先登录获取 Token
TOKEN=$(curl -s -X POST http://localhost:8101/api/user/login \
  -H "Content-Type: application/json" \
  -d '{"userAccount":"test","userPassword":"12345678"}' \
  | jq -r '.data.token')

# 测量提交接口响应时间（重复 10 次取平均）
for i in $(seq 1 10); do
  curl -s -o /dev/null -w "响应时间: %{time_total}s\n" \
    -X POST http://localhost:8101/api/question_submit/do \
    -H "Content-Type: application/json" \
    -H "Authorization: Bearer $TOKEN" \
    -d '{
      "questionId": 1,
      "language": "java",
      "code": "import java.util.Scanner;\npublic class Main {\n  public static void main(String[] args) {\n    Scanner sc = new Scanner(System.in);\n    int a = sc.nextInt(), b = sc.nextInt();\n    System.out.println(a + b);\n  }\n}"
    }'
  sleep 0.5
done
```

### 预期结果

| 模式 | 接口响应时间 |
|------|-------------|
| 同步（判题在请求线程中执行） | 1-10 秒（取决于代码执行时间） |
| 异步（RabbitMQ，仅入库+发消息） | 10-50 毫秒 |

**提升倍数**：约 100-1000 倍

## 测试 2：并发吞吐量压测

### 目的

测试高并发下系统的处理能力。

### 工具安装

```bash
# 安装 Apache Bench（ab）
# Ubuntu/Debian
apt-get install apache2-utils

# CentOS
yum install httpd-tools

# 或者用 wrk（更强大）
apt-get install wrk
```

### 用 ab 压测

```bash
# 先准备请求文件
cat > /tmp/submit_request.json << 'EOF'
{
  "questionId": 1,
  "language": "java",
  "code": "import java.util.Scanner;\npublic class Main {\n  public static void main(String[] args) {\n    Scanner sc = new Scanner(System.in);\n    int a = sc.nextInt(), b = sc.nextInt();\n    System.out.println(a + b);\n  }\n}"
}
EOF

# 获取 Token
TOKEN=$(curl -s -X POST http://localhost:8101/api/user/login \
  -H "Content-Type: application/json" \
  -d '{"userAccount":"test","userPassword":"12345678"}' \
  | jq -r '.data.token')

# 压测：100 个并发，共 500 个请求
ab -n 500 -c 100 \
  -p /tmp/submit_request.json \
  -T "application/json" \
  -H "Authorization: Bearer $TOKEN" \
  http://localhost:8101/api/question_submit/do

# 用 wrk 压测（更精确）
wrk -t4 -c100 -d30s \
  -s /tmp/submit.lua \
  http://localhost:8101/api/question_submit/do
```

### wrk 脚本（/tmp/submit.lua）

```lua
wrk.method = "POST"
wrk.headers["Content-Type"] = "application/json"
wrk.headers["Authorization"] = "Bearer YOUR_TOKEN_HERE"
wrk.body = '{"questionId":1,"language":"java","code":"import java.util.Scanner;\npublic class Main {\n  public static void main(String[] args) {\n    Scanner sc = new Scanner(System.in);\n    int a = sc.nextInt(), b = sc.nextInt();\n    System.out.println(a + b);\n  }\n}"}'
```

### 预期结果

| 并发数 | 同步模式 QPS | 异步模式 QPS | 提升 |
|--------|-------------|-------------|------|
| 10 | ~1-5 | ~100-200 | 20-100x |
| 50 | 超时/报错 | ~300-500 | — |
| 100 | 服务器崩溃 | ~500-800 | — |

## 测试 3：队列积压与消化速度

### 目的

测试突发流量下队列的缓冲能力。

### 方法

1. 短时间发送大量请求（模拟高峰期）
2. 观察队列积压数量
3. 测量消化完所有消息的时间

```bash
# 1. 快速发送 100 个请求
for i in $(seq 1 100); do
  curl -s -o /dev/null \
    -X POST http://localhost:8101/api/question_submit/do \
    -H "Content-Type: application/json" \
    -H "Authorization: Bearer $TOKEN" \
    -d '{"questionId":1,"language":"python","code":"print(sum(map(int, input().split())))"}' &
done
wait

# 2. 查看队列积压
# 访问 RabbitMQ 管理界面：http://localhost:15672
# 用户名/密码：guest/guest
# 查看 oj.judge.queue 的 Messages Ready 数量

# 3. 监控消化速度
watch -n 1 'curl -s -u guest:guest http://localhost:15672/api/queues/%2F/oj.judge.queue | jq ".messages"'
```

### 预期结果

| 阶段 | 队列积压 | 说明 |
|------|----------|------|
| 发送完 100 个请求 | ~100 条 | 消息在队列中排队 |
| 消费中（prefetch=1） | 逐渐减少 | 每个判题任务约 2-5 秒 |
| 全部消化完 | 0 条 | 约 200-500 秒（100 × 2-5 秒） |

**关键点**：异步模式下，100 个请求在几秒内全部返回成功，用户不用等待判题完成。

## 测试 4：失败重试验证

### 目的

验证手动 ACK + 死信队列的可靠性。

### 方法

1. 模拟消费者异常（手动停止后端）
2. 发送请求
3. 重启后端，观察消息是否被重新消费

```bash
# 1. 停止后端
docker compose stop backend

# 2. 发送 5 个请求（会失败，但消息已在队列中）
for i in $(seq 1 5); do
  curl -s -X POST http://localhost:8101/api/question_submit/do \
    -H "Content-Type: application/json" \
    -H "Authorization: Bearer $TOKEN" \
    -d '{"questionId":1,"language":"python","code":"print(1)"}' || true
done

# 3. 查看队列中的消息数
curl -s -u guest:guest http://localhost:15672/api/queues/%2F/oj.judge.queue | jq ".messages"

# 4. 重启后端
docker compose up -d backend

# 5. 观察日志，确认消息被消费
docker compose logs -f backend
```

### 预期结果

- 停止后端期间，消息在队列中等待
- 重启后端后，消息被自动消费，判题完成
- 无消息丢失

## 测试 5：死信队列兜底

### 目的

验证判题失败时消息进入死信队列。

### 方法

1. 提交一个会判题失败的代码（如编译错误）
2. 观察消息是否进入死信队列

```bash
# 提交编译错误的代码
curl -s -X POST http://localhost:8101/api/question_submit/do \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $TOKEN" \
  -d '{
    "questionId": 1,
    "language": "java",
    "code": "this is not valid java code"
  }'

# 查看死信队列
curl -s -u guest:guest http://localhost:15672/api/queues/%2F/oj.judge.queue.dlq | jq ".messages"
```

### 预期结果

- 编译错误的代码，判题结果为 FAILED（不会进入死信队列，因为这是正常业务失败）
- 只有消费者异常（如 OOM、网络中断）才会触发 `basicNack`，消息进入死信队列
