# 第 9 章 数据库设计

## 本章目标

- 理解 OJ 系统的数据库表结构设计
- 掌握 JSON 字段在 MySQL 中的使用场景
- 理解软删除 vs 硬删除的设计决策
- 了解索引策略

---

## 9.1 数据库概述

- **数据库**：MySQL 8.0
- **数据库名**：`yuoj`
- **字符集**：`utf8mb4`（支持 Emoji 和特殊字符）
- **排序规则**：`utf8mb4_unicode_ci`

### MyBatis-Plus 配置

```yaml
mybatis-plus:
  configuration:
    map-underscore-to-camel-case: false  # 关闭驼峰映射
  global-config:
    db-config:
      logic-delete-field: isDelete      # 逻辑删除字段
      logic-delete-value: 1             # 已删除
      logic-not-delete-value: 0         # 未删除
      id-type: auto                     # 自增主键
```

## 9.2 表结构详解

### user 表（用户）

```sql
CREATE TABLE `user` (
  `id`            BIGINT       NOT NULL AUTO_INCREMENT COMMENT '用户ID',
  `userAccount`   VARCHAR(256) NOT NULL COMMENT '账号',
  `userPassword`  VARCHAR(512) NOT NULL COMMENT '密码（MD5加密）',
  `userName`      VARCHAR(256) DEFAULT NULL COMMENT '用户名',
  `userAvatar`    VARCHAR(1024) DEFAULT NULL COMMENT '头像URL',
  `userProfile`   VARCHAR(512) DEFAULT NULL COMMENT '个人简介',
  `userRole`      VARCHAR(256) NOT NULL DEFAULT 'user' COMMENT '角色：user/admin/ban',
  `createTime`    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updateTime`    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  `isDelete`      TINYINT      NOT NULL DEFAULT 0 COMMENT '是否删除：0-未删除，1-已删除',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_userAccount` (`userAccount`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='用户表';
```

**字段说明**：
- `userRole`：三种角色 — `user`（普通用户）、`admin`（管理员）、`ban`（封禁）
- `userPassword`：存储 BCrypt 加密后的密码（旧数据为 MD5，登录时优先 BCrypt 失败回退 MD5）

### question 表（题目）

```sql
CREATE TABLE `question` (
  `id`            BIGINT        NOT NULL AUTO_INCREMENT COMMENT '题目ID',
  `title`         VARCHAR(512)  DEFAULT NULL COMMENT '标题',
  `content`       TEXT          DEFAULT NULL COMMENT '内容（Markdown）',
  `tags`          VARCHAR(1024) DEFAULT NULL COMMENT '标签（JSON数组）',
  `answer`        TEXT          DEFAULT NULL COMMENT '参考答案',
  `judgeCase`     TEXT          DEFAULT NULL COMMENT '测试用例（JSON数组）',
  `judgeConfig`   VARCHAR(1024) DEFAULT NULL COMMENT '判题配置（JSON）',
  `submitNum`     INT           NOT NULL DEFAULT 0 COMMENT '提交次数',
  `acceptedNum`   INT           NOT NULL DEFAULT 0 COMMENT '通过次数',
  `userId`        BIGINT        NOT NULL COMMENT '创建者ID',
  `createTime`    DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `updateTime`    DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  `isDelete`      TINYINT       NOT NULL DEFAULT 0,
  PRIMARY KEY (`id`),
  KEY `idx_userId` (`userId`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='题目表';
```

**JSON 字段示例**：

```json
// tags
["数组", "哈希表", "双指针", "排序"]

// judgeCase
[
  {"input": "[2,7,11,15]\n9", "output": "[0,1]"},
  {"input": "[3,2,4]\n6", "output": "[1,2]"},
  {"input": "[3,3]\n6", "output": "[0,1]"}
]

// judgeConfig
{"timeLimit": 1000, "memoryLimit": 256, "stackLimit": 128}
```

### question_submit 表（提交记录）

```sql
CREATE TABLE `question_submit` (
  `id`            BIGINT        NOT NULL AUTO_INCREMENT COMMENT '提交ID',
  `language`      VARCHAR(128)  NOT NULL COMMENT '编程语言',
  `code`          TEXT          NOT NULL COMMENT '用户代码',
  `judgeInfo`     VARCHAR(1024) DEFAULT NULL COMMENT '判题结果（JSON）',
  `status`        INT           NOT NULL DEFAULT 0 COMMENT '状态：0-待判题，1-判题中，2-成功，3-失败',
  `questionId`    BIGINT        NOT NULL COMMENT '题目ID',
  `userId`        BIGINT        NOT NULL COMMENT '用户ID',
  `createTime`    DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `updateTime`    DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  `isDelete`      TINYINT       NOT NULL DEFAULT 0,
  PRIMARY KEY (`id`),
  KEY `idx_questionId` (`questionId`),
  KEY `idx_userId` (`userId`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='题目提交表';
```

**judgeInfo JSON 示例**：

```json
// 通过
{"message": "Accepted", "time": 15, "memory": 10240}

// 失败
{"message": "Wrong Answer", "time": 10, "memory": 8192}

// 编译错误
{"message": "Compilation Error", "time": 0, "memory": 0}
```

## 9.3 设计决策分析

### 软删除 vs 硬删除

| 表 | 删除方式 | 原因 |
|---|----------|------|
| `user` | **软删除** | 用户数据珍贵，封禁后可恢复 |
| `question` | **软删除** | 题目有提交记录关联，不能物理删除 |
| `question_submit` | **软删除** | 提交记录需要保留，用于统计分析 |

**软删除的实现**：
```java
@TableName("question")
public class Question {
    @TableLogic  // MyBatis-Plus 逻辑删除注解
    private Integer isDelete;  // 0=未删除, 1=已删除
}

// 调用 deleteById 时，MyBatis-Plus 自动生成：
// UPDATE question SET isDelete = 1 WHERE id = ?
// 而不是：DELETE FROM question WHERE id = ?
```

### JSON 字段的使用

| 字段 | 类型 | 使用场景 |
|------|------|----------|
| `question.tags` | JSON 数组 | 灵活的标签系统，支持多标签 |
| `question.judgeCase` | JSON 数组 | 测试用例列表，结构化存储 |
| `question.judgeConfig` | JSON 对象 | 判题配置（时间/内存/栈限制）|
| `question_submit.judgeInfo` | JSON 对象 | 判题结果（时间/内存/消息）|

**JSON 字段的优势**：
1. 灵活性：不需要为每个属性创建单独的列
2. 结构化：比纯文本更易解析
3. 查询支持：MySQL 8.0 支持 JSON 函数查询

**JSON 字段的限制**：
1. 索引效率低：无法直接对 JSON 内部字段建索引
2. 查询复杂：需要使用 `JSON_EXTRACT` 等函数
3. 更新成本高：修改 JSON 内部字段需要重写整个字段

### 索引策略

| 表 | 索引 | 类型 | 用途 |
|---|------|------|------|
| `user` | `uk_userAccount` | 唯一 | 登录查询 |
| `question` | `idx_userId` | 普通 | 按创建者查询 |
| `question_submit` | `idx_questionId` | 普通 | 按题目查询提交 |
| `question_submit` | `idx_userId` | 普通 | 按用户查询提交 |

## 9.4 小结

| 要点 | 内容 |
|------|------|
| **表数量** | 3 张核心表 |
| **JSON 字段** | tags、judgeCase、judgeConfig、judgeInfo |
| **软删除** | 主表用 isDelete，关联表用硬删除 |
| **索引** | 外键字段建普通索引，防重复字段建唯一索引 |
| **字符集** | utf8mb4_unicode_ci |
