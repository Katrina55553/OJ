## 8.1 判题全流程概览

```
用户提交代码
    │
    ▼
QuestionController.doQuestionSubmit()
    │
    ▼
QuestionSubmitServiceImpl.submitQuestionSubmit()
    ├── 参数校验（题目是否存在、语言是否支持）
    ├── 入库（status = WAITING = 0）
    └── 发送消息到 RabbitMQ（oj.judge.queue）
    │
    ▼
┌─────────────────────────────────────┐
│           RabbitMQ 消息队列          │
│        队列：oj.judge.queue         │
│        模式：手动 ACK               │
│        prefetch: 1                  │
└─────────────────────────────────────┘
    │
    ▼
JudgeMessageConsumer.onMessage()
    │
    ▼
JudgeServiceImpl.doJudge(submitId)
    ├── 查询提交记录
    ├── 更新状态为 RUNNING（1）
    └── 调用判题核心
    │
    ▼
JudgeManager.doJudge(questionSubmit)
    ├── CodeSandboxFactory.getSandbox() → DockerCodeSandbox
    ├── 获取题目信息和测试用例
    └── 调用沙箱执行代码
    │
    ▼
DockerCodeSandbox.executeCode(request)
    ├── 创建临时目录，写入用户代码
    ├── docker build 构建语言镜像
    └── 对每个测试用例：
        ├── docker run 创建容器（6 层安全限制）
        ├── stdin 输入测试数据
        ├── 后台线程监控内存（docker stats）
        ├── 读取 stdout/stderr
        └── 超时则销毁容器
    │
    ▼
JudgeStrategy.judge(judgeContext)
    ├── 逐测试用例比对输出
    └── 返回 JudgeInfo（time, memory, message）
    │
    ▼
更新数据库
    ├── status = SUCCEED（2）或 FAILED（3）
    └── judgeInfo = { time, memory, message }
```

## 8.2 CodeSandbox 接口设计

### 接口定义

```java
public interface CodeSandbox {
    /**
     * 执行代码
     * @param request 包含代码、语言、输入数据
     * @return 包含输出、错误信息、执行时间、内存使用
     */
    ExecuteCodeResponse executeCode(ExecuteCodeRequest request);
}
```

### 请求/响应模型

```java
@Data
public class ExecuteCodeRequest {
    private String code;          // 用户代码
    private String language;      // 编程语言
    private List<String> inputs;  // 测试用例输入列表
}

@Data
public class ExecuteCodeResponse {
    private List<String> outputs;   // 每个测试用例的输出
    private String message;         // 错误信息
    private Integer status;         // 0=成功, 1=编译错误, 2=运行错误, 3=超时
    private Long time;              // 执行时间（ms）
    private Long memory;            // 内存使用（KB）
}
```

### 三种实现

| 实现类 | 说明 | 状态 |
|--------|------|------|
| **DockerCodeSandbox** | Docker 容器隔离执行 | ✅ 当前使用 |
| RemoteCodeSandbox | 远程沙箱服务（localhost:8090） | 可用 |
| ThirdPartyCodeSandbox | 第三方 API（JDoodle） | 可用 |

### CodeSandboxFactory

```java
public class CodeSandboxFactory {
    public static CodeSandbox getSandbox(String type) {
        switch (type) {
            case "docker":
                return new DockerCodeSandbox();
            case "remote":
                return new RemoteCodeSandbox();
            case "thirdparty":
                return new ThirdPartyCodeSandbox();
            default:
                return new DockerCodeSandbox();
        }
    }
}
```

## 8.3 DockerCodeSandbox 实现（核心）

### DooD 架构

本项目采用 **Docker-outside-of-Docker（DooD）** 模式：

```
┌─────────────────────────────────────────┐
│  主机 Docker Engine                     │
│                                         │
│  ┌─────────────────────────────────┐    │
│  │  oj-backend 容器                │    │
│  │  ┌─────────────────────────┐   │    │
│  │  │  DockerCodeSandbox      │   │    │
│  │  │  (Java 代码)            │   │    │
│  │  └────────────┬────────────┘   │    │
│  │               │                 │    │
│  │  docker build / docker run     │    │
│  │  (通过挂载的 Docker Socket)    │    │
│  └───────────────┼─────────────────┘    │
│                  │                       │
│  ┌───────────────┼─────────────────┐    │
│  │  /var/run/docker.sock           │    │
│  └───────────────┼─────────────────┘    │
│                  │                       │
│  ┌───────────────┼─────────────────┐    │
│  │  沙箱容器 1   │  沙箱容器 2     │    │
│  │  (用户代码)   │  (用户代码)     │    │
│  └───────────────┴─────────────────┘    │
└─────────────────────────────────────────┘
```

**关键配置**：
- 后端容器挂载 Docker Socket：`/var/run/docker.sock:/var/run/docker.sock`
- 后端容器内安装 Docker CLI：`apt-get install -y docker.io`
- 沙箱 Dockerfile 挂载：`./sandbox:/app/sandbox`

### 执行流程详解

```java
@Value("${codesandbox.docker.workDir:/app/sandbox/tmp}")
private String workDirBase;  // 必须建在宿主机与后端容器共享的卷上

@Override
public ExecuteCodeResponse executeCode(ExecuteCodeRequest request) {
    String code = request.getCode();
    String language = request.getLanguage();
    List<String> inputs = request.getInputs();

    // 1. 在共享卷上创建工作目录（关键！DooD 架构下宿主机 Docker 才能看到）
    Path baseDir = Paths.get(workDirBase);
    Files.createDirectories(baseDir);
    Path workDir = Files.createTempDirectory(baseDir, "oj-sandbox-");

    try {
        // 2. 写入用户代码到文件
        String codeFilename = getCodeFilename(language);
        Path codeFile = workDir.resolve(codeFilename);
        Files.write(codeFile, code.getBytes(StandardCharsets.UTF_8));

        // 设置 POSIX 权限（755 目录 / 644 文件），确保 --user=nobody 可读
        Set<PosixFilePermission> perms = new HashSet<>();
        perms.add(PosixFilePermission.OWNER_READ);
        perms.add(PosixFilePermission.OWNER_WRITE);
        perms.add(PosixFilePermission.OWNER_EXECUTE);
        perms.add(PosixFilePermission.GROUP_READ);
        perms.add(PosixFilePermission.GROUP_EXECUTE);
        perms.add(PosixFilePermission.OTHERS_READ);
        perms.add(PosixFilePermission.OTHERS_EXECUTE);
        Files.setPosixFilePermissions(workDir, perms);

        // 3. 确保基础镜像存在（预构建，不每次都 build）
        String baseImage = "oj-base-" + language;
        ensureBaseImage(language, baseImage);

        // 4. 编译型语言先编译（C++/Java/Go）
        if (needsCompilation(language)) {
            compileCodeInContainer(baseImage, workDir, language);
        }

        // 5. 对每个测试用例执行
        List<String> outputs = new ArrayList<>();
        long totalTime = 0;
        long totalMemory = 0;

        for (String input : inputs) {
            ContainerResult result = runContainerWithMount(baseImage, workDir, language, input);
            outputs.add(result.stdout);
            totalTime += result.execTime;
            totalMemory += result.memoryUsage;

            if (result.exitCode != 0) {
                // 运行错误，直接返回
                return buildErrorResponse(result);
            }
        }

        // 6. 构建成功响应
        return ExecuteCodeResponse.builder()
                .outputList(outputs)
                .status(0)
                .judgeInfo(JudgeInfo.builder()
                        .message("Accepted")
                        .time(totalTime)
                        .memory(totalMemory / 1024)
                        .build())
                .build();

    } finally {
        // 7. 清理临时文件
        deleteDirectory(workDir);
    }
}
```

> **关键设计：workDir 必须建在共享卷上**
>
> 后端容器通过 docker socket 调宿主机 Docker（DooD 架构），如果用 `Files.createTempDirectory("oj-sandbox-")` 默认建在容器内 `/tmp`，宿主机 Docker 看不到该路径，`docker run -v containerPath:/code` 会挂载一个空目录，导致沙箱容器内找不到用户代码。
>
> 解决方案：workDir 建在 docker-compose 已挂载的共享卷 `./oj-backend/sandbox:/app/sandbox` 上，容器内 `/app/sandbox/tmp/xxx` 对应宿主机 `./oj-backend/sandbox/tmp/xxx`，`docker run -v` 才能找到文件。

### 安全隔离（6 层防护）

```java
private ContainerResult runContainerWithMount(String baseImage, Path workDir,
        String language, String input) {
    String containerName = "oj-run-" + UUID.randomUUID().toString().substring(0, 8);

    List<String> cmd = new ArrayList<>();
    cmd.add("docker");
    cmd.add("run");
    cmd.add("--name=" + containerName);

    // ===== 第 1 层：内存限制（防 OOM 攻击）=====
    cmd.add("--memory=" + memoryLimit);        // 256m

    // ===== 第 2 层：CPU 限制（防 CPU 耗尽）=====
    cmd.add("--cpus=" + cpuLimit);             // 1

    // ===== 第 3 层：禁止网络（防数据外泄、作弊）=====
    cmd.add("--network=none");

    // ===== 第 4 层：进程数限制（防 fork bomb）=====
    cmd.add("--pids-limit=50");

    // ===== 第 5 层：只读文件系统（防文件攻击）=====
    cmd.add("--read-only");

    // ===== 第 6 层：非 root 用户（限制权限提升）=====
    cmd.add("--user=nobody");

    // 挂载 workDir 到沙箱容器的 /code（只读）
    cmd.add("-v");
    cmd.add(workDir.toString() + ":/code:ro");

    // 保持 stdin 打开，传入测试数据
    cmd.add("-i");
    cmd.add(baseImage);
    cmd.add("bash");
    cmd.add("-c");
    cmd.add(getRunCommand(language));  // 如 "python3 solution.py"

    // 执行命令...
}
```

### 6 层防护详解

| 层 | Docker 参数 | 防护目标 | 攻击示例 |
|---|-------------|----------|----------|
| 1 | `--memory=256m` | 防止内存耗尽 | `int arr[999999999];` |
| 2 | `--cpus=1` | 防止 CPU 耗尽 | `while(true){}` |
| 3 | `--network=none` | 防止网络访问 | `requests.get("http://evil.com")` |
| 4 | `--pids-limit=50` | 防止 fork bomb | `while(true){fork();}` |
| 5 | `--read-only` | 防止文件攻击 | `rm -rf /` |
| 6 | `--user=nobody` | 防止权限提升 | `sudo su` |

### 内存监控

```java
private long monitorMemory(String containerName) {
    AtomicLong maxMemory = new AtomicLong(0);

    // 后台线程轮询 docker stats
    Thread monitor = new Thread(() -> {
        while (true) {
            try {
                // 调用 docker stats 获取内存使用
                ProcessBuilder pb = new ProcessBuilder(
                    "docker", "stats", containerName,
                    "--no-stream",
                    "--format={{.MemUsage}}"
                );
                Process process = pb.start();
                BufferedReader reader = new BufferedReader(
                    new InputStreamReader(process.getInputStream())
                );
                String line = reader.readLine();
                if (line != null) {
                    // 解析 "15.5MiB / 256MiB" 格式
                    long memory = parseMemoryUsage(line);
                    maxMemory.updateAndGet(current -> Math.max(current, memory));
                }
                Thread.sleep(500); // 每 500ms 采样一次
            } catch (Exception e) {
                break; // 容器已退出
            }
        }
    });
    monitor.setDaemon(true);
    monitor.start();

    return maxMemory.get();
}
```

## 8.4 JudgeStrategy 判题策略

### 策略模式

```java
public interface JudgeStrategy {
    JudgeInfo judge(JudgeContext context);
}
```

### JudgeContext

```java
@Data
public class JudgeContext {
    private Question question;           // 题目信息
    private QuestionSubmit questionSubmit; // 提交记录
    private ExecuteCodeResponse executeCodeResponse; // 沙箱执行结果
    private List<JudgeCase> judgeCases;  // 测试用例列表
}
```

### DefaultJudgeStrategy

```java
@Component
public class DefaultJudgeStrategy implements JudgeStrategy {
    @Override
    public JudgeInfo judge(JudgeContext context) {
        List<String> expectedOutputs = context.getJudgeCases().stream()
            .map(JudgeCase::getOutput)
            .collect(Collectors.toList());

        List<String> actualOutputs = context.getExecuteCodeResponse().getOutputs();

        // 逐测试用例比对
        JudgeInfo judgeInfo = new JudgeInfo();
        judgeInfo.setMessage("Accepted");
        judgeInfo.setMemory(context.getExecuteCodeResponse().getMemory());
        judgeInfo.setTime(context.getExecuteCodeResponse().getTime());

        for (int i = 0; i < expectedOutputs.size(); i++) {
            String expected = expectedOutputs.get(i).trim();
            String actual = actualOutputs.get(i).trim();

            if (!expected.equals(actual)) {
                judgeInfo.setMessage("Wrong Answer");
                return judgeInfo;
            }
        }

        return judgeInfo;
    }
}
```

### JavaLanguageJudgeStrategy

Java 语言有特殊要求：**公共类名必须与文件名一致**（`Solution.java` 中的 `class Solution`）。

```java
@Component
public class JavaLanguageJudgeStrategy implements JudgeStrategy {
    @Override
    public JudgeInfo judge(JudgeContext context) {
        // 1. 检查代码是否包含 "class Solution"
        String code = context.getQuestionSubmit().getCode();
        if (!code.contains("class Solution")) {
            JudgeInfo judgeInfo = new JudgeInfo();
            judgeInfo.setMessage("Compilation Error: 类名必须为 Solution");
            return judgeInfo;
        }

        // 2. 委托给默认策略进行输出比对
        return new DefaultJudgeStrategy().judge(context);
    }
}
```

## 8.5 RabbitMQ 异步解耦

### 为什么需要消息队列

```
同步判题的问题：

1. 阻塞：用户提交后需要等待判题完成，可能等待 10 秒以上
2. 超时：HTTP 请求超时导致判题中断
3. 并发：100 个用户同时提交，后端同时创建 100 个容器，资源耗尽
4. 耦合：提交逻辑和判题逻辑强耦合

异步解耦的好处：

1. 即时响应：提交后立即返回提交 ID，用户可以查看其他页面
2. 削峰填谷：MQ 缓冲突发流量，后端按能力消费
3. 重试机制：判题失败自动重试
4. 解耦：提交和判题独立，互不影响
```

### 生产者（JudgeMessageProducer）

```java
@Component
public class JudgeMessageProducer {

    @Autowired
    private RabbitTemplate rabbitTemplate;

    // 发送判题消息
    public void sendMessage(Long questionSubmitId) {
        rabbitTemplate.convertAndSend(
            "oj.judge.exchange",    // 交换机
            "oj.judge.routing",    // 路由键
            questionSubmitId        // 消息内容（提交 ID）
        );
    }
}
```

### 消费者（JudgeMessageConsumer）

```java
@Component
public class JudgeMessageConsumer {

    @Autowired
    private JudgeService judgeService;

    @RabbitListener(
        queues = "oj.judge.queue",
        ackMode = "MANUAL"  // 手动 ACK
    )
    public void onMessage(Long questionSubmitId, Message message,
                          Channel channel) throws IOException {
        try {
            // 1. 调用判题服务
            judgeService.doJudge(questionSubmitId);

            // 2. 手动确认消息
            channel.basicAck(message.getMessageProperties().getDeliveryTag(), false);
        } catch (Exception e) {
            // 3. 判题失败，拒绝消息并重新入队
            channel.basicNack(
                message.getMessageProperties().getDeliveryTag(),
                false,
                true  // 重新入队
            );
        }
    }
}
```

### RabbitMQ 配置

```yaml
spring:
  rabbitmq:
    host: rabbitmq
    port: 5672
    username: guest
    password: guest
    listener:
      simple:
        acknowledge-mode: manual  # 手动 ACK
        prefetch: 1               # 一次只消费一条
        retry:
          enabled: true
          max-attempts: 3         # 最多重试 3 次
          initial-interval: 3000  # 首次重试间隔 3 秒
```

### 判题状态流转

```
WAITING (0)  ──→  RUNNING (1)  ──→  SUCCEED (2)
                       │
                       └──→  FAILED (3)

状态说明：
  0 = WAITING：等待判题（刚提交）
  1 = RUNNING：判题中（Consumer 开始处理）
  2 = SUCCEED：判题通过（所有测试用例通过）
  3 = FAILED：判题失败（有测试用例不通过或运行错误）
```

## 8.6 JudgeManager 协调器

```java
@Component
public class JudgeManager {

    @Autowired
    private CodeSandboxFactory codeSandboxFactory;

    @Autowired
    private JudgeStrategy defaultJudgeStrategy;

    @Autowired
    private JudgeStrategy javaLanguageJudgeStrategy;

    public JudgeInfo doJudge(QuestionSubmit questionSubmit, Question question) {
        // 1. 获取沙箱实例
        CodeSandbox sandbox = codeSandboxFactory.getSandbox("docker");

        // 2. 构建执行请求
        ExecuteCodeRequest request = new ExecuteCodeRequest();
        request.setCode(questionSubmit.getCode());
        request.setLanguage(questionSubmit.getLanguage());
        request.setInputs(parseJudgeCases(question.getJudgeCase()));

        // 3. 调用沙箱执行
        ExecuteCodeResponse response = sandbox.executeCode(request);

        // 4. 选择判题策略
        JudgeStrategy strategy = "java".equals(questionSubmit.getLanguage())
            ? javaLanguageJudgeStrategy
            : defaultJudgeStrategy;

        // 5. 构建判题上下文
        JudgeContext context = new JudgeContext();
        context.setQuestion(question);
        context.setQuestionSubmit(questionSubmit);
        context.setExecuteCodeResponse(response);
        context.setJudgeCases(parseJudgeCases(question.getJudgeCase()));

        // 6. 执行判题策略
        return strategy.judge(context);
    }
}
```

## 8.8 镜像复用与定时清理

### 性能优化：镜像复用（-v 挂载）

8.3 节提到 `ensureBaseImage` 预构建基础镜像，这里展开其背后的性能优化思路。

**最初的实现（每次 docker build）**：

```
用户提交代码 → 写入临时文件 → docker build（打包镜像）→ docker run
```

**问题**：每次 `docker build` 约 5-15 秒，判题大部分时间花在构建镜像上。

**优化后的实现（基础镜像 + -v 挂载）**：把"编译器/运行时环境"与"用户代码"分开，环境只构建一次，代码每次通过 -v 挂载进去。

**第 1 步：预先构建基础镜像**（只包含编译器/运行时，不含用户代码）

```dockerfile
# sandbox/Dockerfile.base.cpp
FROM m.daocloud.io/docker.io/library/gcc:latest
WORKDIR /code
```

```dockerfile
# sandbox/Dockerfile.base.java
FROM m.daocloud.io/docker.io/library/eclipse-temurin:17-jdk
WORKDIR /code
```

构建一次，永久复用：

```bash
docker build -t oj-base-cpp -f Dockerfile.base.cpp .
docker build -t oj-base-java -f Dockerfile.base.java .
```

**第 2 步：运行时 -v 挂载用户代码**

```
用户提交代码 → 写入 /tmp/sandbox_xxx/solution.cpp
                     │
                     ▼
          docker run --rm \
            --memory=256m --cpus=1 \
            --network=none --pids-limit=50 \
            --read-only --user=nobody \
            -v /tmp/sandbox_xxx:/code:ro \    # 关键：把用户代码目录挂载进容器
            -i oj-base-cpp \
            bash -c "g++ -o solution solution.cpp -O2 && ./solution"
```

**第 3 步：编译型语言（C++/Java/Go）— 先编译再运行**

```java
// DockerCodeSandbox.java 核心逻辑
if (needsCompilation(language)) {
    // 先在容器内编译一次（仍用 -v 挂载，把编译后的二进制写回宿主目录）
    docker run -v workDir:/code oj-base-cpp bash -c "g++ -o solution solution.cpp -O2";
}

// 对每个测试用例，直接用已编译好的二进制执行
for (String input : inputList) {
    docker run -v workDir:/code:ro -i oj-base-cpp bash -c "./solution";
}
```

### 性能对比

| 阶段 | 旧方案（每次 build） | 新方案（-v 挂载） | 提升 |
|------|----------------------|-------------------|------|
| **首次执行** | 15-20 秒（build + run） | 10-15 秒（build 基础镜像 + 挂载） | 首次略慢 |
| **重复执行** | 15-20 秒/次 | 1-3 秒/次（无需 build） | **5-10 倍** |
| **磁盘占用** | 每次产生新镜像，需定期清理 | 基础镜像固定，无残留 | **显著降低** |

### DockerCleanupTask 定时清理任务

即便 `docker run` 带了 `--rm`，异常退出（超时 kill、OOM）的容器仍可能残留。长期运行会占满磁盘。新增 `DockerCleanupTask`（@Scheduled 定时任务）：

- **每小时**：清理已停止的容器（`docker container prune -f`）
- **每 6 小时**：清理悬空镜像（`docker image prune -f`）
- **每天**：删除 `oj-sandbox-*` 临时构建产物镜像（保留 `oj-base-*` 基础镜像）

```java
@Component
@Slf4j
public class DockerCleanupTask {

    /**
     * 每小时清理已停止的容器
     */
    @Scheduled(cron = "0 0 * * * ?")
    public void cleanupStoppedContainers() {
        try {
            ProcessBuilder pb = new ProcessBuilder("docker", "container", "prune", "-f");
            Process process = pb.start();
            int exitCode = process.waitFor();
            log.info("清理已停止容器完成，exitCode={}", exitCode);
        } catch (Exception e) {
            log.error("清理容器失败", e);
        }
    }

    /**
     * 每 6 小时清理悬空镜像（dangling images）
     */
    @Scheduled(cron = "0 0 */6 * * ?")
    public void cleanupDanglingImages() {
        try {
            ProcessBuilder pb = new ProcessBuilder("docker", "image", "prune", "-f");
            Process process = pb.start();
            int exitCode = process.waitFor();
            log.info("清理悬空镜像完成，exitCode={}", exitCode);
        } catch (Exception e) {
            log.error("清理悬空镜像失败", e);
        }
    }

    /**
     * 每天删除 oj-sandbox-* 临时构建产物镜像（保留 oj-base-* 基础镜像）
     */
    @Scheduled(cron = "0 0 3 * * ?")
    public void cleanupTempImages() {
        try {
            // 列出所有 oj-sandbox-* 镜像
            ProcessBuilder listPb = new ProcessBuilder(
                "docker", "images", "--format={{.Repository}}:{{.Tag}}",
                "--filter=reference=oj-sandbox-*"
            );
            Process listProcess = listPb.start();
            BufferedReader reader = new BufferedReader(
                new InputStreamReader(listProcess.getInputStream())
            );
            String line;
            while ((line = reader.readLine()) != null) {
                // 逐个删除临时镜像（基础镜像 oj-base-* 不受影响）
                ProcessBuilder delPb = new ProcessBuilder("docker", "rmi", "-f", line.trim());
                Process delProcess = delPb.start();
                delProcess.waitFor();
                log.info("删除临时镜像: {}", line.trim());
            }
        } catch (Exception e) {
            log.error("清理临时镜像失败", e);
        }
    }
}
```

> **为什么需要清理？** 高并发场景下，每个测试用例产生一个容器，即便有 `--rm`，异常退出的容器仍可能残留，长期运行会占满磁盘。

## 8.9 死信队列与可靠性兜底

8.5 节的消费者在判题异常时使用 `basicNack(..., requeue=true)` 让消息重新入队。但如果消息反复失败，会陷入无限重试。死信队列（Dead Letter Queue, DLQ）是更可靠的兜底方案。

### 什么是死信队列？

```
正常队列：oj.judge.queue
    │
    │ 消息被 reject / 超时 / 队列满了
    ▼
死信队列：oj.judge.queue.dlq
    │
    ▼
死信消费者：记录日志、告警、人工处理
```

### 死信队列配置

通过 `QueueBuilder` 为正常队列绑定死信路由，并设置消息 TTL：

```java
@Configuration
public class RabbitMQConfig {

    // 正常队列
    @Bean
    public Queue judgeQueue() {
        return QueueBuilder.durable("oj.judge.queue")
            .withArgument("x-dead-letter-exchange", "")           // 死信交换机（默认）
            .withArgument("x-dead-letter-routing-key", "oj.judge.queue.dlq")  // 死信路由
            .withArgument("x-message-ttl", 30000)                 // 消息 TTL：30 秒
            .build();
    }

    // 死信队列
    @Bean
    public Queue deadLetterQueue() {
        return QueueBuilder.durable("oj.judge.queue.dlq").build();
    }
}
```

### 消息什么时候进入死信队列？

| 触发条件 | 说明 |
|----------|------|
| `basicNack` + `requeue=false` | 消费者拒绝消息，不重新入队 |
| 消息 TTL 过期 | 消息在队列中超过 30 秒没被消费 |
| 队列满了 | 队列达到最大长度 |

### 死信消费者

死信消费者负责记录日志、更新提交状态为失败，保证不会有提交永远卡在 WAITING 状态：

```java
@RabbitListener(queues = "oj.judge.queue.dlq")
public void onDeadLetter(Message message, Channel channel) throws IOException {
    String body = new String(message.getBody());
    log.error("判题消息进入死信队列: submitId={}", body);

    // 更新提交状态为失败
    long submitId = Long.parseLong(body);
    QuestionSubmit update = new QuestionSubmit();
    update.setId(submitId);
    update.setStatus(QuestionSubmitStatusEnum.FAILED.getValue());
    questionSubmitService.updateById(update);

    channel.basicAck(message.getMessageProperties().getDeliveryTag(), false);
}
```

> **作用**：异常兜底，保证不会有一个提交永远卡在 WAITING 状态。

### 手动 ACK vs 自动 ACK

8.5 节使用手动 ACK（`acknowledge-mode: manual`），这里对比两种模式的差异：

| ACK 模式 | 行为 | 风险 |
|----------|------|------|
| **自动 ACK** | 消费者收到消息就确认 | 如果消费者崩溃，消息丢失 |
| **手动 ACK** | 消费者处理完才确认 | 消费者崩溃，消息回到队列，重新投递 |

**场景**：消费者收到消息，开始判题，执行到一半崩溃了。
- 自动 ACK：消息已确认，丢失了，这个提交永远没有结果
- 手动 ACK：消息没确认，回到队列，其他消费者重新处理

### 消息完整生命周期

```
1. 用户提交 → 发送消息到 oj.judge.queue
2. 消费者收到消息 → 手动 ACK → 判题成功 → 结束
3. 消费者收到消息 → 判题异常 → basicNack(requeue=false) → 进入死信队列
4. 死信消费者记录日志 → 更新状态为失败 → 人工排查
```

## 8.10 消息可靠性三环节与积压处理

### 消息可靠性：三个环节保障

消息从生产到消费要经过三个环节，每个环节都可能丢消息，需要分别保障：

```
生产者 ──→ 队列 ──→ 消费者
  │          │         │
  │          │         └─ 环节 3：手动 ACK（处理完才确认）
  │          └─ 环节 2：队列持久化（durable=true）
  └─ 环节 1：生产者 confirm（到达交换机回调）
```

#### 环节 1：生产者 → 队列（publisher confirm）

8.5 节的 `JudgeMessageProducer` 通过 `rabbitTemplate.convertAndSend()` 发送消息。RabbitMQ 的 `publisher confirm` 机制确保消息到达交换机后回调 `confirmCallback`，如果发送失败会触发回调通知。

**保障**：消息从生产者到队列的传输不丢失。

#### 环节 2：队列本身（消息持久化）

8.9 节的 `RabbitMQConfig` 中通过 `QueueBuilder.durable("oj.judge.queue")` 声明持久化队列：

```java
return QueueBuilder.durable("oj.judge.queue")  // ⭐ 持久化队列
    .withArgument("x-dead-letter-exchange", "")
    .withArgument("x-dead-letter-routing-key", "oj.judge.queue.dlq")
    .withArgument("x-message-ttl", 30000)
    .build();
```

**保障**：`durable=true`，队列元数据持久化到磁盘，RabbitMQ 重启不丢。

#### 环节 3：消费者（手动 ACK）

8.5 节的消费者使用手动 ACK，关键在于"处理完才确认"：

```java
channel.basicAck(deliveryTag, false);      // ⭐ 成功才确认
channel.basicNack(deliveryTag, false, false);  // ⭐ 失败拒绝，不重新入队（进死信）
```

**保障**：消费者崩溃时，未确认的消息自动回到队列，不会丢失。

### 消息积压怎么处理？

#### 什么是消息积压？

```
生产速度 >> 消费速度：

每秒提交 100 个判题任务
每个判题任务需要 3 秒
每秒只能消费 1 个（prefetch=1）

→ 每秒积压 99 个消息
→ 队列越来越大
→ 用户等待时间越来越长
```

#### 解决方案

| 方案 | 原理 | 适用场景 |
|------|------|----------|
| **增加消费者** | 多开几个消费者并行处理 | 最直接有效 |
| **增加 prefetch** | 每个消费者同时处理多条消息 | CPU 够用时 |
| **临时扩容** | 高峰期多开消费者，低峰期缩容 | 有自动扩缩容能力 |
| **丢弃过期消息** | 设置 TTL，超时直接进死信 | 允许部分任务超时 |

#### 本项目的处理

当前配置为单消费者、`prefetch=1`，适合中小规模。如果要支持高并发：
1. 水平扩展消费者（多部署几个后端实例）
2. 增大 prefetch 到 3-5
3. 考虑用线程池并行处理

### 消息重复消费与幂等

**为什么会重复消费？**
- 消费者处理完但 ACK 之前崩溃 → 消息回到队列 → 重新投递给其他消费者
- 网络抖动导致 ACK 丢失 → 消息被重新投递

**本项目的处理**：目前没有做幂等处理，理论上手动 ACK + prefetch=1 不会重复消费。如果要做幂等，可以在消费前检查提交状态：如果已经不是 WAITING 状态，直接跳过。

```java
@RabbitListener(queues = "oj.judge.queue")
public void onMessage(Message message, Channel channel) throws IOException {
    long deliveryTag = message.getMessageProperties().getDeliveryTag();
    try {
        long questionSubmitId = Long.parseLong(new String(message.getBody()));

        // 幂等检查：如果状态已经不是 WAITING，说明已处理过，直接 ACK
        QuestionSubmit submit = questionSubmitService.getById(questionSubmitId);
        if (!QuestionSubmitStatusEnum.WAITING.getValue().equals(submit.getStatus())) {
            channel.basicAck(deliveryTag, false);
            return;
        }

        judgeService.doJudge(questionSubmitId);
        channel.basicAck(deliveryTag, false);
    } catch (Exception e) {
        channel.basicNack(deliveryTag, false, false);
    }
}
```
