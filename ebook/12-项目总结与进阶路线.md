## 12.1 架构亮点回顾

### DooD 沙箱架构

本项目最核心的架构亮点是 **Docker-outside-of-Docker（DooD）** 代码沙箱：

```
后端容器（Spring Boot）
    │
    │  /var/run/docker.sock（挂载）
    │
    ▼
宿主机 Docker Engine
    │
    ├── 沙箱容器 1（用户 A 的 C++ 代码）
    ├── 沙箱容器 2（用户 B 的 Python 代码）
    └── 沙箱容器 3（用户 C 的 Java 代码）
```

**设计优势**：
1. **安全隔离**：6 层防护（内存/CPU/网络/PID/文件系统/用户）
2. **语言独立**：每种语言独立 Dockerfile，互不影响
3. **资源可控**：每个容器独立资源限制
4. **易于扩展**：新增语言只需添加 Dockerfile

### 异步判题解耦

```
同步模式（不可取）：
  用户提交 → 等待 10 秒 → 返回结果
  问题：阻塞、超时、并发瓶颈

异步模式（本项目）：
  用户提交 → 立即返回提交 ID → MQ 缓冲 → 后台判题
  优势：即时响应、削峰填谷、重试机制
```

### 双状态管理

```
Vuex（用户认证）          Pinia（主题切换）
  │                         │
  │  项目初始化时使用        │  新功能使用
  │  功能稳定               │  API 更简洁
  │  不需要替换             │  Composition API 风格
  │                         │
  └──────── 共存 ───────────┘
```

**渐进式迁移策略**：新功能用 Pinia，老功能保持 Vuex，逐步迁移。

## 12.2 设计模式应用

### 策略模式（Strategy）

**应用场景**：判题策略选择

```java
// 策略接口
public interface JudgeStrategy {
    JudgeInfo judge(JudgeContext context);
}

// 默认策略
@Component("defaultJudgeStrategy")
public class DefaultJudgeStrategy implements JudgeStrategy { ... }

// Java 特殊策略
@Component("javaLanguageJudgeStrategy")
public class JavaLanguageJudgeStrategy implements JudgeStrategy { ... }

// 使用
JudgeStrategy strategy = "java".equals(language)
    ? javaLanguageJudgeStrategy
    : defaultJudgeStrategy;
strategy.judge(context);
```

**优势**：新增语言的特殊判题逻辑，只需添加新策略类，不修改现有代码。

### 工厂模式（Factory）

**应用场景**：沙箱实例创建

```java
public class CodeSandboxFactory {
    public static CodeSandbox getSandbox(String type) {
        switch (type) {
            case "docker":    return new DockerCodeSandbox();
            case "remote":    return new RemoteCodeSandbox();
            case "thirdparty": return new ThirdPartyCodeSandbox();
            default:          return new DockerCodeSandbox();
        }
    }
}
```

**优势**：客户端无需知道具体实现类，通过配置切换沙箱类型。

### 模板方法模式（Template Method）

**应用场景**：DockerCodeSandbox 执行流程

```java
public ExecuteCodeResponse executeCode(ExecuteCodeRequest request) {
    // 1. 创建临时目录        ← 固定步骤
    // 2. 写入用户代码        ← 固定步骤
    // 3. 复制 Dockerfile     ← 固定步骤
    // 4. 构建 Docker 镜像    ← 固定步骤
    // 5. 执行测试用例        ← 固定步骤（但每语言不同）
    // 6. 监控资源使用        ← 固定步骤
    // 7. 清理临时文件        ← 固定步骤
}
```

**优势**：算法骨架固定，子类可以覆盖特定步骤。

### 观察者模式（Observer）

**应用场景**：主题切换的系统偏好监听

```typescript
// 监听系统主题变化
window.matchMedia("(prefers-color-scheme: dark)")
  .addEventListener("change", (e) => {
    if (!localStorage.getItem("theme")) {
      isDark.value = e.matches;
      applyTheme();
    }
  });
```

### 代理模式（Proxy）

**应用场景**：Nginx 反向代理

```
用户浏览器 → Nginx(:3000) → 后端(:8101)
              │
              ├── 静态资源：直接返回
              ├── /api/*：反向代理
              └── SSE：WebSocket 升级
```

## 12.3 可改进方向

### 安全加固

| 问题 | 改进方案 |
|------|----------|
| JWT 密钥硬编码 | 使用环境变量注入，生产环境必须修改 |
| 密码使用 MD5 | 升级为 BCrypt（Spring Security） |
| SQL 注入风险 | MyBatis-Plus 参数化查询已防护，但仍需审查 |
| XSS 防护 | Markdown 渲染已禁用 `html: true`，需全面审查 |

### 性能优化

| 问题 | 改进方案 |
|------|----------|
| 数据库索引 | 分析慢查询日志，添加缺失索引 |
| 缓存策略 | 增加题目详情缓存、热门题目缓存 |
| 前端懒加载 | 路由组件使用 `() => import()` 动态导入 |
| 图片优化 | 使用 WebP 格式、CDN 加速 |

### 代码质量

| 问题 | 改进方案 |
|------|----------|
| 单元测试覆盖 | 增加 Service 层单元测试 |
| 代码规范 | 统一 ESLint + Prettier 规则 |
| 类型安全 | 前端 TypeScript 严格模式 |

### 功能完善

| 功能 | 现状 | 改进 |
|------|------|------|
| 用户资料编辑 | 前端 mock | 对接后端修改接口 |
| 邮件通知 | 未实现 | 判题结果邮件通知 |

### 运维改进

| 问题 | 改进方案 |
|------|----------|
| 日志收集 | 统一日志收集与分析 |
| 监控告警 | Prometheus + Grafana |
| CI/CD | GitHub Actions 自动构建部署 |

## 12.6 结语

### 全栈项目的价值

通过这个 OJ 项目，你已经接触到了全栈开发的核心技术：

```
前端：Vue 3 + TypeScript + 组件设计 + 状态管理 + 权限控制
后端：Spring Boot + 分层架构 + JWT 认证 + AOP + MQ
数据库：MySQL + JSON 字段 + 软删除 + 索引设计
DevOps：Docker + Compose + Nginx + 多阶段构建
核心：Docker 沙箱 + 安全隔离 + 异步判题
```

技术在不断发展，保持学习的习惯：

1. **读源码**：Spring Boot、Vue 3 的源码是最好的教材

3. **写博客**：输出倒逼输入，写技术文章加深理解
4. **参与开源**：贡献代码、提 Issue、写文档
