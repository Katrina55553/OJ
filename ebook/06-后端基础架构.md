## 6.1 项目结构与分层

### 分层架构

```
请求 → Controller → Service → Manager → Mapper → 数据库
         │            │          │         │
         │            │          │         └── 数据访问（MyBatis-Plus）
         │            │          └── 可复用业务能力封装
         │            └── 核心业务逻辑
         └── REST 接口（参数校验、响应封装）
```

| 层 | 职责 | 示例 |
|---|------|------|
| **Controller** | REST 接口，参数校验，调用 Service | `UserController.userLogin()` |
| **Service** | 核心业务逻辑 | `UserService.login()` |
| **Manager** | 可复用的业务能力封装 | `Manager.executeCode()` |
| **Mapper** | 数据库访问（MyBatis-Plus） | `UserMapper.selectById()` |
| **Model** | 数据模型 | Entity / DTO / VO / Enum |

### Model 子包

| 包 | 用途 | 示例 |
|---|------|------|
| `entity` | 数据库实体，与表一一对应 | `User.java` |
| `dto` | 请求参数对象（前端 → 后端） | `UserLoginRequest.java` |
| `vo` | 响应视图对象（后端 → 前端，脱敏） | `UserVO.java` |
| `enums` | 业务枚举 | `UserRoleEnum.java` |

## 6.4 JWT 认证流程

### JWT 基础

**JWT（JSON Web Token）** 是一种无状态的认证方案，由三部分组成：

```
Header.Payload.Signature

Header:    {"alg": "HS256", "typ": "JWT"}  → Base64 编码
Payload:   {"userId": 1, "exp": 1700000000} → Base64 编码
Signature: HMAC-SHA256(header + payload, secret) → 签名
```

本项目配置：
- **算法**：HS256（对称加密）
- **过期时间**：604800000ms = **7 天**
- **密钥**：通过 `JWT_SECRET` 环境变量注入

### 登录流程

```java
// UserController.java
@PostMapping("/login")
public BaseResponse<LoginUserVO> userLogin(@RequestBody UserLoginRequest request,
                                            HttpServletRequest httpRequest) {
    // 1. 参数校验
    if (request == null || StringUtils.isBlank(request.getUserAccount())
        || StringUtils.isBlank(request.getUserPassword())) {
        return ResultUtils.error(ErrorCode.PARAMS_ERROR);
    }

    // 2. 调用 Service 登录
    LoginUserVO loginUserVO = userService.userLogin(request, httpRequest);
    return ResultUtils.success(loginUserVO);
}

// UserServiceImpl.java
@Override
public LoginUserVO userLogin(UserLoginRequest request, HttpServletRequest httpRequest) {
    String account = request.getUserAccount();
    String password = request.getUserPassword();

    // 1. 查询用户
    QueryWrapper<User> queryWrapper = new QueryWrapper<>();
    queryWrapper.eq("userAccount", account);
    User user = userMapper.selectOne(queryWrapper);
    if (user == null) {
        throw new BusinessException(ErrorCode.PARAMS_ERROR, "账号不存在");
    }

    // 2. 校验密码（优先 BCrypt，失败回退 MD5 兼容旧数据）
    if (!PASSWORD_ENCODER.matches(password, user.getUserPassword())) {
        // 旧数据是 MD5，尝试 MD5 比对后异步升级为 BCrypt
        if (!DigestUtils.md5DigestAsHex(password.getBytes()).equals(user.getUserPassword())) {
            throw new BusinessException(ErrorCode.PARAMS_ERROR, "密码错误");
        }
    }

    // 3. 生成 JWT
    String token = JwtUtils.createToken(user.getId());

    // 4. 返回脱敏用户信息
    return getSafetyUser(user, token);
}
```

### JWT 工具类

```java
public class JwtUtils {
    @Value("${jwt.secret}")
    private static String secret;

    @Value("${jwt.expiration}")
    private static Long expiration; // 604800000ms = 7 天

    // 生成 Token
    public static String createToken(Long userId) {
        return JWT.create()
            .withClaim("userId", userId)
            .withExpiresAt(new Date(System.currentTimeMillis() + expiration))
            .sign(Algorithm.HMAC256(secret));
    }

    // 解析 Token
    public static Long parseToken(String token) {
        DecodedJWT jwt = JWT.require(Algorithm.HMAC256(secret))
            .build()
            .verify(token);
        return jwt.getClaim("userId").asLong();
    }
}
```

### JwtInterceptor

```java
@Component
public class JwtInterceptor implements HandlerInterceptor {

    @Override
    public boolean preHandle(HttpServletRequest request,
                             HttpServletResponse response,
                             Object handler) {
        // 1. 获取 Authorization 头
        String token = request.getHeader("Authorization");
        if (StringUtils.isBlank(token) || !token.startsWith("Bearer ")) {
            throw new BusinessException(ErrorCode.NOT_LOGIN_ERROR);
        }
        token = token.substring(7); // 去掉 "Bearer " 前缀

        // 2. 解析 Token
        Long userId;
        try {
            userId = JwtUtils.parseToken(token);
        } catch (JWTVerificationException e) {
            throw new BusinessException(ErrorCode.NOT_LOGIN_ERROR, "Token 无效或已过期");
        }

        // 3. 查询用户信息（优先从 Redis 缓存）
        String cacheKey = "user:id:" + userId;
        User user = (User) redisTemplate.opsForValue().get(cacheKey);
        if (user == null) {
            user = userMapper.selectById(userId);
            if (user != null) {
                redisTemplate.opsForValue().set(cacheKey, user, 30, TimeUnit.MINUTES);
            }
        }
        if (user == null) {
            throw new BusinessException(ErrorCode.NOT_LOGIN_ERROR);
        }

        // 4. 设置用户上下文到 Request 属性
        request.setAttribute("loginUser", user);

        return true;
    }
}
```

## 6.5 权限校验（AOP）

### @AuthCheck 注解

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface AuthCheck {
    String mustRole() default "";  // 必须具有的角色
}
```

### AuthInterceptor（AOP 切面）

```java
@Aspect
@Component
public class AuthInterceptor {

    @Around("@annotation(authCheck)")
    public Object doInterceptor(ProceedingJoinPoint joinPoint,
                                 AuthCheck authCheck) throws Throwable {
        // 1. 获取当前用户
        HttpServletRequest request = ((ServletRequestAttributes)
            RequestContextHolder.getRequestAttributes()).getRequest();
        User loginUser = (User) request.getAttribute("loginUser");

        // 2. 获取注解要求的角色
        String mustRole = authCheck.mustRole();
        UserRoleEnum mustRoleEnum = UserRoleEnum.getEnumByValue(mustRole);

        // 3. 如果不需要特定角色，放行
        if (mustRoleEnum == null) {
            return joinPoint.proceed();
        }

        // 4. 获取用户角色
        UserRoleEnum userRoleEnum = UserRoleEnum.getEnumByValue(loginUser.getUserRole());
        if (userRoleEnum == null) {
            throw new BusinessException(ErrorCode.NO_AUTH_ERROR);
        }

        // 5. 管理员可以访问所有接口
        if (userRoleEnum.equals(UserRoleEnum.ADMIN)) {
            return joinPoint.proceed();
        }

        // 6. 角色不匹配，拒绝
        if (!userRoleEnum.equals(mustRoleEnum)) {
            throw new BusinessException(ErrorCode.NO_AUTH_ERROR);
        }

        return joinPoint.proceed();
    }
}
```

### 使用示例

```java
// 只有管理员可以添加题目
@PostMapping("/add")
@AuthCheck(mustRole = "admin")
public BaseResponse<Long> addQuestion(@RequestBody QuestionAddRequest request) {
    // ...
}

// 只有管理员可以删除题目
@PostMapping("/delete")
@AuthCheck(mustRole = "admin")

    // ...

```

### 三种角色

| 角色 | 值 | 权限 |
|------|-----|------|
| `USER` | `"user"` | 普通用户：刷题、提交、查看记录 |
| `ADMIN` | `"admin"` | 管理员：题目管理、用户管理、全部权限 |
| `BAN` | `"ban"` | 封禁用户：无法登录或使用系统 |

## 6.6 接口限流

### @RateLimit 注解

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface RateLimit {
    int count() default 10;      // 时间窗口内允许的请求数
    int time() default 60;       // 时间窗口（秒）
}
```

### RateLimitInterceptor

```java
@Component
public class RateLimitInterceptor implements HandlerInterceptor {

    @Autowired
    private RedisTemplate<String, Integer> redisTemplate;

    @Override
    public boolean preHandle(HttpServletRequest request,
                             HttpServletResponse response,
                             Object handler) {
        // 1. 获取方法上的 @RateLimit 注解
        if (!(handler instanceof HandlerMethod)) return true;
        HandlerMethod handlerMethod = (HandlerMethod) handler;
        RateLimit rateLimit = handlerMethod.getMethodAnnotation(RateLimit.class);
        if (rateLimit == null) return true;

        // 2. 构建限流 key（IP + 接口路径）
        String ip = request.getRemoteAddr();
        String uri = request.getRequestURI();
        String key = "rate_limit:" + ip + ":" + uri;

        // 3. Redis 滑动窗口计数
        Long count = redisTemplate.opsForValue().increment(key);
        if (count == 1) {
            // 首次请求，设置过期时间
            redisTemplate.expire(key, rateLimit.time(), TimeUnit.SECONDS);
        }

        // 4. 超过限制，拒绝请求
        if (count > rateLimit.count()) {
            throw new BusinessException(ErrorCode.SYSTEM_ERROR, "请求过于频繁");
        }

        return true;
    }
}
```

### 使用示例

```java

@RateLimit(count = 5, time = 60)  // 每分钟最多 5 次登录尝试

    // ...
```

## 6.7 日志拦截器

```java
@Component
public class LogInterceptor implements HandlerInterceptor {

    private static final Logger log = LoggerFactory.getLogger(LogInterceptor.class);
    private final ThreadLocal<Long> startTime = new ThreadLocal<>();

    @Override
    public boolean preHandle(HttpServletRequest request, ...) {
        startTime.set(System.currentTimeMillis());
        log.info("Request: {} {} from {}",
            request.getMethod(),
            request.getRequestURI(),
            request.getRemoteAddr());
        return true;
    }

    @Override
    public void afterCompletion(HttpServletRequest request, ...) {
        long cost = System.currentTimeMillis() - startTime.get();
        log.info("Response: {} {} cost {}ms",
            request.getMethod(),
            request.getRequestURI(),
            cost);
        startTime.remove(); // 防止内存泄漏
    }
}
```

## 6.9 滑动窗口限流与 Lua 脚本

6.6 节给出了基于 `INCR + EXPIRE` 的固定窗口实现，简单但存在窗口边界突发问题（两个窗口交界处可能通过 2 倍请求）。本节介绍更精确的**滑动窗口 + Lua 脚本**方案。

### 限流算法对比

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **固定窗口** | 每 N 秒一个窗口，窗口内计数 | 简单 | 窗口边界突发（两个窗口交界处可能通过 2 倍请求） |
| **滑动窗口** | 窗口随时间滑动，精确计数 | 精确 | 需要更多存储 |
| **令牌桶** | 固定速率往桶里放令牌，请求消耗令牌 | 允许一定突发 | 实现复杂 |
| **漏桶** | 请求进入桶，固定速率流出 | 平滑 | 不允许突发 |

**本项目使用滑动窗口**，因为：
1. 实现简单（Redis + Lua 几行代码）
2. 精确（不存在固定窗口的边界问题）
3. 性能好（Redis 单线程，原子操作）

### 滑动窗口原理

```
时间轴（60 秒窗口，限制 5 次请求）：

    ←───── 60 秒窗口 ─────→
    |                       |
    ▼                       ▼
────┬───┬───┬───┬───┬───┬───┬───→ 时间
    t1  t2  t3  t4  t5  t6  t7

    t1: 请求 1 → 计数 1 ✓
    t2: 请求 2 → 计数 2 ✓
    t3: 请求 3 → 计数 3 ✓
    t4: 请求 4 → 计数 4 ✓
    t5: 请求 5 → 计数 5 ✓
    t6: 请求 6 → 计数 6 ✗（超过限制，拒绝）

    当 t7 时，窗口滑动，t1 过期，计数变为 5
```

### Redis ZSET 实现

```
Key: rate_limit:192.168.1.1:/api/user/login
Value: Sorted Set
  - 成员：请求的唯一 ID（UUID 或时间戳）
  - 分数：请求的时间戳（毫秒）
```

**每次请求**：
1. 删除窗口外的旧记录（`ZREMRANGEBYSCORE`）
2. 统计窗口内的记录数（`ZCARD`）
3. 如果超过限制，拒绝
4. 否则，添加当前请求记录（`ZADD`）

### 完整 Lua 脚本

```lua
-- rate_limit.lua
-- KEYS[1]：限流 key（如 rate_limit:ip:uri）
-- ARGV[1]：窗口大小（毫秒）
-- ARGV[2]：最大请求数
-- ARGV[3]：当前时间戳（毫秒）
-- ARGV[4]：请求唯一标识

-- 1. 删除窗口外的旧记录
redis.call('ZREMRANGEBYSCORE', KEYS[1], 0, ARGV[3] - ARGV[1])

-- 2. 统计窗口内的请求数
local current = redis.call('ZCARD', KEYS[1])

-- 3. 判断是否超过限制
if current < tonumber(ARGV[2]) then
    -- 未超过，添加当前请求
    redis.call('ZADD', KEYS[1], ARGV[3], ARGV[4])
    -- 设置 key 过期时间（窗口大小 + 1 秒，避免残留）
    redis.call('PEXPIRE', KEYS[1], ARGV[1])
    return 1  -- 允许
else
    return 0  -- 拒绝
end
```

### 为什么用 Lua 脚本

**不用 Lua**（有并发问题）：
```
线程 A: 读取计数 = 4
线程 B: 读取计数 = 4
线程 A: 计数 4 < 5，允许，写入计数 = 5
线程 B: 计数 4 < 5，允许，写入计数 = 5
结果：窗口内通过了 6 个请求！
```

**用 Lua**（原子操作）：
```
Lua 脚本在 Redis 中是原子执行的
线程 A 和线程 B 的整个"删除-判断-写入"过程不会被打断
```

> Redis 的"删除-判断-写入"是三步操作。如果用 Java 代码，多线程并发时会出现竞态条件（两个线程同时读到计数 4，都认为可以放行）。Lua 脚本在 Redis 中原子执行，不会被打断。

### AOP @Around 切面执行 Lua

```java
@Aspect
@Component
public class RateLimitInterceptor {

    @Resource
    private StringRedisTemplate redisTemplate;

    @Around("@annotation(rateLimit)")
    public Object around(ProceedingJoinPoint joinPoint, RateLimit rateLimit) throws Throwable {
        // 1. 构造限流 key
        HttpServletRequest request = ((ServletRequestAttributes)
            RequestContextHolder.getRequestAttributes()).getRequest();
        String ip = request.getRemoteAddr();
        String uri = request.getRequestURI();
        String key = "rate_limit:" + rateLimit.key() + ":" + ip + ":" + uri;

        // 2. 执行 Lua 脚本
        Long result = redisTemplate.execute(
            rateLimitScript,                          // Lua 脚本
            Collections.singletonList(key),           // KEYS
            String.valueOf(rateLimit.windowSeconds() * 1000),  // 窗口大小（毫秒）
            String.valueOf(rateLimit.count()),         // 最大请求数
            String.valueOf(System.currentTimeMillis()), // 当前时间
            UUID.randomUUID().toString()               // 请求唯一标识
        );

        // 3. 判断结果
        if (result == null || result == 0) {
            throw new BusinessException(ErrorCode.SYSTEM_ERROR, "请求过于频繁");
        }

        // 4. 放行
        return joinPoint.proceed();
    }
}
```

限流 key 设计为 `rate_limit:{业务}:{IP}:{URI}`，三个维度区分：业务（登录/注册/提交）、IP（不同用户）、URI（不同接口）。

## 6.10 Token 黑名单与双 Token 方案

### Session vs JWT 传统对比

**传统 Session 模式**：

```
用户登录
    │
    ▼
后端创建 Session，存入内存/Redis
    │  SessionId = abc123
    ▼
返回 Cookie: SessionId=abc123
    │
    ▼
后续请求携带 Cookie
    │
    ▼
后端根据 SessionId 查找用户信息
```

**问题**：
- 服务端需要存储 Session（内存/Redis）
- 分布式部署需要 Session 共享
- Cookie 有跨域限制

**JWT 模式（本项目）**：

```
用户登录
    │
    ▼
后端生成 JWT Token（包含用户信息 + 签名）
    │  eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOjF9.xxx
    ▼
返回 Token（存在响应体中）
    │
    ▼
前端存入 localStorage
    │
    ▼
后续请求 Header: Authorization: Bearer <Token>
    │
    ▼
后端解析 Token，直接获取用户信息（无需查库）
```

**JWT 优势**：
- 无状态：服务端不存储任何会话信息
- 无跨域问题：Token 放在 Header 中，不依赖 Cookie
- 易扩展：多服务器部署无需 Session 共享

### Token 黑名单（jti 方案）

JWT 一旦签发，在过期前都有效，**无法主动失效**（用户修改密码后，旧 Token 仍然有效）。解决方案是维护一个 Token 黑名单。

**Redis + jti（JWT ID）方案**：

1. **生成 Token 时**，添加 `jti`（UUID）作为唯一标识
2. **退出登录时**，将 `blacklist:token:{jti}` 写入 Redis，TTL = Token 剩余有效期
3. **每次请求时**，拦截器检查 Token 的 jti 是否在 Redis 黑名单中，如果在则返回 401

**Redis Key 设计**：
```
Key:   blacklist:token:{jti}
Value: "1"（只需 key 存在）
TTL:   Token 剩余有效期（毫秒）
```

**为什么用 jti 而不是存整个 Token？**
- jti 是固定的 UUID 字符串（36 字符），Token 是变长的（可能几百字符）
- jti 更省 Redis 存储空间
- 同一个用户多次登录会生成不同 jti，可以精确失效某一次登录的 Token

### 双 Token 方案（静默刷新）

普通方案中 Token 过期后用户需要重新登录，体验差。更高级的方案是**双 Token（AccessToken + RefreshToken）**：

| Token | 有效期 | 用途 |
|-------|--------|------|
| **AccessToken** | 短期（2 小时） | 业务请求鉴权，放在 `Authorization` 头 |
| **RefreshToken** | 长期（7 天） | 仅用于 AccessToken 过期时静默刷新 |

**流程**：
1. 登录时同时返回 AccessToken 和 RefreshToken
2. 前端正常请求带 AccessToken
3. AccessToken 过期收到 401 时，前端用 RefreshToken 调用 `/refresh` 接口换取新的 AccessToken
4. 整个过程对用户透明（静默刷新），无需重新登录
5. RefreshToken 也过期时才跳转登录页

> 前端收到 401 响应后，清除 localStorage 中的 Token，跳转到登录页。更高级的方案是用双 Token（AccessToken + RefreshToken），AccessToken 短期（2 小时），RefreshToken 长期（7 天），AccessToken 过期时用 RefreshToken 静默刷新。
