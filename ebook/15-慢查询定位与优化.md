# 第 15 章 慢查询定位与优化

## 面试怎么答

> "定位慢查询分三步：开启慢查询日志 → 分析 EXPLAIN 执行计划 → 针对性优化。常见优化手段包括添加索引、优化 SQL 写法、避免 SELECT *、减少子查询。"

---

## 1. 开启慢查询日志

### MySQL 配置

```sql
-- 查看当前配置
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';

-- 开启慢查询日志
SET GLOBAL slow_query_log = ON;

-- 设置阈值（超过 1 秒记录）
SET GLOBAL long_query_time = 1;

-- 查看日志位置
SHOW VARIABLES LIKE 'slow_query_log_file';
```

### Docker 环境下的日志位置

```bash
# 进入 MySQL 容器
docker compose exec mysql bash

# 查看慢查询日志
cat /var/lib/mysql/*.slow.log

# 或者直接查
docker compose exec mysql mysql -uroot -p123456 -e "SHOW VARIABLES LIKE 'slow_query_log_file';"
```

## 2. EXPLAIN 分析执行计划

### 用法

```sql
EXPLAIN SELECT * FROM question WHERE title LIKE '%算法%';
```

### 输出字段详解

| 字段 | 含义 | 重点关注 |
|------|------|----------|
| **type** | 训问类型 | `ALL`（全表扫描）最差，`const`（常量查找）最好 |
| **key** | 实际使用的索引 | `NULL` 表示没用索引 |
| **rows** | 预估扫描行数 | 越小越好 |
| **Extra** | 额外信息 | `Using filesort`、`Using temporary` 需要优化 |

### type 访问类型（从好到差）

```
system > const > eq_ref > ref > range > index > ALL

const  → 主键或唯一索引等值查询（SELECT * FROM user WHERE id = 1）
ref    → 普通索引等值查询（SELECT * FROM question WHERE userId = 1）
range  → 索引范围查询（SELECT * FROM question WHERE id > 100）
index  → 全索引扫描（比 ALL 好，但还是要扫描整个索引）
ALL    → 全表扫描（最差，必须优化）
```

### 示例

```sql
-- ❌ 全表扫描（type=ALL）
EXPLAIN SELECT * FROM question WHERE title LIKE '%算法%';
-- title 没有索引，MySQL 只能一行一行扫

-- ✅ 使用索引（type=ref）
EXPLAIN SELECT * FROM question WHERE userId = 1;
-- userId 有索引 idx_userId，直接定位

-- ❌ 索引失效（type=ALL）
EXPLAIN SELECT * FROM question WHERE YEAR(createTime) = 2024;
-- 对索引列使用函数，索引失效
```

## 3. 常见慢查询场景与优化

### 场景 1：没加索引

```sql
-- ❌ 慢查询
SELECT * FROM question_submit WHERE userId = 123;
-- userId 没有索引，全表扫描

-- ✅ 优化：添加索引
ALTER TABLE question_submit ADD INDEX idx_userId(userId);
```

### 场景 2：SELECT *

```sql
-- ❌ 查所有字段
SELECT * FROM question WHERE userId = 1;
-- 即使有索引，也要回表取所有字段

-- ✅ 只查需要的字段
SELECT id, title, content FROM question WHERE userId = 1;
-- 覆盖索引，不需要回表
```

### 场景 3：索引失效

```sql
-- ❌ 对索引列使用函数
SELECT * FROM question WHERE YEAR(createTime) = 2024;
-- 索引失效，全表扫描

-- ✅ 改写为范围查询
SELECT * FROM question WHERE createTime >= '2024-01-01'
  AND createTime < '2025-01-01';

-- ❌ 隐式类型转换
SELECT * FROM user WHERE userAccount = 12345;
-- userAccount 是 varchar，传入 int，触发隐式转换，索引失效

-- ✅ 类型匹配
SELECT * FROM user WHERE userAccount = '12345';

-- ❌ LIKE 左模糊
SELECT * FROM question WHERE title LIKE '%算法%';
-- 左边有 %，索引失效

-- ✅ 右模糊可以用索引
SELECT * FROM question WHERE title LIKE '算法%';
```

### 场景 4：子查询慢

```sql
-- ❌ 子查询
SELECT * FROM question_submit
WHERE userId IN (SELECT id FROM user WHERE userRole = 'admin');

-- ✅ 改写为 JOIN
SELECT qs.* FROM question_submit qs
INNER JOIN user u ON qs.userId = u.id
WHERE u.userRole = 'admin';
```

### 场景 5：ORDER BY 没有索引

```sql
-- ❌ filesort（Extra 中出现 Using filesort）
SELECT * FROM question ORDER BY createTime DESC;

-- ✅ 添加索引
ALTER TABLE question ADD INDEX idx_createTime(createTime);
```

### 场景 6：分页偏移量大

```sql
-- ❌ 深分页（offset 越大越慢）
SELECT * FROM question ORDER BY id LIMIT 1000000, 20;
-- MySQL 需要扫描前 1000020 行，丢弃前 1000000 行

-- ✅ 游标分页（记住上次的 id）
SELECT * FROM question WHERE id > 1000000 ORDER BY id LIMIT 20;
-- 直接从 id=1000000 开始取，跳过扫描
```

## 4. 本项目中的索引设计

```sql
-- user 表
UNIQUE KEY uk_userAccount (userAccount)  -- 登录查询
-- login: SELECT * FROM user WHERE userAccount = ? AND userPassword = ?

-- question 表
KEY idx_userId (userId)                  -- 按创建者查询
-- listMy: SELECT * FROM question WHERE userId = ?

-- question_submit 表
KEY idx_questionId (questionId)          -- 按题目查询提交
KEY idx_userId (userId)                  -- 按用户查询提交
-- listSubmit: SELECT * FROM question_submit WHERE questionId = ?
```

## 5. 面试常见追问

### Q: 怎么发现慢查询？

> 1. **慢查询日志**：MySQL 开启 `slow_query_log`，记录超过阈值的 SQL
> 2. **SHOW PROCESSLIST**：查看当前正在执行的 SQL
> 3. **APM 工具**：SkyWalking、Pinpoint 等自动采集慢 SQL

### Q: 索引越多越好吗？

> 不是。索引的代价：
> 1. **写入变慢**：每次 INSERT/UPDATE/DELETE 都要维护索引
> 2. **占用空间**：索引也是数据结构（B+ 树），占磁盘
> 3. **可能不被使用**：MySQL 优化器可能选择不用索引
>
> 建议：只为高频查询的 WHERE、ORDER BY、JOIN 条件建索引。

### Q: 什么时候索引会失效？

> 1. 对索引列使用函数（`YEAR(createTime)`）
> 2. 隐式类型转换（varchar 列用 int 查询）
> 3. LIKE 左模糊（`%keyword`）
> 4. OR 条件中有非索引列
> 5. 使用 `!=` 或 `NOT IN`（部分场景）
> 6. 数据量小时，优化器选择全表扫描

### Q: EXPLAIN 中的 Using filesort 和 Using temporary 什么意思？

> - **Using filesort**：MySQL 无法利用索引排序，需要额外排序操作（性能差）
> - **Using temporary**：MySQL 需要创建临时表来处理查询（通常出现在 GROUP BY、DISTINCT）
> - 这两个都出现在 Extra 列，看到就要考虑优化
