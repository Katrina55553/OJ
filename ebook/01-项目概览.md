## 1.1 什么是在线判题系统

### 背景

**在线判题系统（Online Judge，简称 OJ）** 是一种自动化代码评测平台，广泛应用于编程竞赛、面试练习和算法学习。知名的 OJ 平台包括：

| 平台 | 特点 |
|------|------|
| **LeetCode** | 面试刷题、每日一题、竞赛排名 |
| **Codeforces** | 国际竞赛、Rating 系统、题解社区 |
| **HUSTOJ** | 开源 OJ、高校教学常用 |
| **洛谷** | 中文社区、题解丰富 |

### 核心功能

一个完整的 OJ 系统需要解决以下核心问题：

1. **题目管理** — 创建、编辑、删除题目，支持 Markdown 富文本和数学公式
2. **代码提交** — 用户选择语言、编写代码、提交判题
3. **自动判题** — 将用户代码在隔离环境中执行，与预期输出比对
4. **安全隔离** — 用户代码可能包含恶意逻辑，必须在沙箱中安全执行
5. **用户系统** — 注册、登录、权限管理、提交记录

### 本项目的定位

本项目是一个 **全栈 OJ 系统**，具备以下特色：

- **Docker 代码沙箱** — 每种语言独立镜像，6 层安全隔离
- **异步判题** — RabbitMQ 消息队列解耦，支持高并发
- **AI 聊天助手** — 基于 Ollama 的 AI 对话，SSE 流式输出
- **暗色主题** — GitHub Dark 风格，跟随系统偏好自动切换
- **一键部署** — Docker Compose 6 个服务，一条命令启动

## 1.2 架构设计

### 整体架构

### 架构分层说明

| 层 | 职责 | 技术 |
|---|------|------|
| **表现层** | 用户界面、交互逻辑 | Vue 3 + Arco Design Vue |
| **接入层** | 静态资源服务、API 反向代理 | Nginx |
| **接口层** | REST API、参数校验、响应封装 | Spring Boot Controller |
| **业务层** | 核心业务逻辑 | Spring Service |
| **能力层** | 可复用的业务能力 | Spring Manager |
| **数据层** | 数据库访问 | MyBatis-Plus Mapper |
| **基础设施** | 数据库、缓存、消息队列 | MySQL + Redis + RabbitMQ |
| **沙箱层** | 代码执行隔离 | Docker 容器 |

## 1.5 核心业务流程

### 用户认证流程

```
用户注册
    │
    ▼
提交账号 + 密码 + 确认密码
    │
    ▼
参数校验 → 账号查重 → 密码加密（MD5） → 入库
    │
    ▼
注册成功，跳转登录

用户登录
    │
    ▼
提交账号 + 密码
    │
    ▼
账号查库 → 密码比对 → 生成 JWT（HS256，7 天过期）
    │
    ▼
返回 Token → 前端存入 localStorage
    │
    ▼
后续请求携带 Authorization: Bearer <token>
    │
    ▼
JwtInterceptor 解析 Token → 校验过期 → 注入用户上下文
```

### 题目提交与判题流程

这是 OJ 系统最核心的流程，采用 **RabbitMQ 异步解耦** 设计：

```
用户编写代码并点击"提交"
    │
    ▼
前端调用 doQuestionSubmitUsingPost(questionId, code, language)
    │
    ▼
QuestionController.doQuestionSubmit()
    │
    ▼
QuestionSubmitServiceImpl.submitQuestionSubmit()
    ├── 1. 参数校验（题目是否存在、语言是否支持）
    ├── 2. 入库（status = WAITING）
    └── 3. 发送消息到 RabbitMQ（oj.judge.queue）
    │
    ▼
┌─────────────────────────────────────┐
│           RabbitMQ 消息队列          │
│        oj.judge.queue               │
│        手动 ACK，prefetch=1          │
└─────────────────────────────────────┘
    │
    ▼
JudgeMessageConsumer.onMessage()
    │
    ▼
JudgeServiceImpl.doJudge(submitId)
    ├── 1. 查询提交记录
    ├── 2. 更新状态为 RUNNING
    └── 3. 调用判题核心
    │
    ▼
JudgeManager.doJudge(questionSubmit)
    ├── 1. CodeSandboxFactory.getSandbox() → DockerCodeSandbox
    ├── 2. 获取题目信息和测试用例
    └── 3. 调用沙箱执行代码
    │
    ▼
DockerCodeSandbox.executeCode(request)
    ├── 1. 创建临时目录，写入用户代码
    ├── 2. docker build 构建语言镜像
    └── 3. 对每个测试用例：
        ├── docker run 创建容器（6 层安全限制）
        ├── stdin 输入测试数据
        ├── 后台线程监控内存（docker stats）
        ├── 读取 stdout/stderr
        └── 超时则销毁容器
    │
    ▼
JudgeStrategy.judge(judgeContext)
    ├── 逐测试用例比对输出
    └── 返回 JudgeInfo（time, memory, message）
    │
    ▼
更新数据库
    ├── status = SUCCEED（全部通过）/ FAILED（有失败）
    └── judgeInfo = { time, memory, message }
    │
    ▼
前端轮询或跳转查看结果
```

### AI 聊天流程

```
用户输入问题
    │
    ▼
前端建立 SSE 连接：GET /api/ai/stream?prompt=xxx
    │
    ▼
AiChatController.streamChat()
    ├── 1. 保存用户消息到 ai_message 表
    └── 2. 调用 Ollama /api/generate（流式）
    │
    ▼
Ollama 逐 token 返回
    │
    ▼
后端通过 SSE 逐 chunk 推送给前端
    │
    ▼
前端 EventSource 接收
    ├── 逐字追加到 AI 消息（打字机动画）
    ├── MdPreview 渲染 Markdown/代码/LaTeX
    └── 自动滚动到底部
    │
    ▼
流结束后保存 AI 完整回复到 ai_message 表
```

| 要点 | 内容 |
